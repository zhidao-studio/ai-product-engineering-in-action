# 码值规范

## 1. 目的

本文件定义数据库、后端、接口和前端之间的码值一致性规则。

码值规范用于解决：

```text
数据库存 1 / 0
后端写 true / false
接口返回 yes / no
前端再翻译一遍
```

这类不一致问题。

## 2. 基本原则

```text
数据库码值是源头。
后端常量必须和数据库码值一致。
接口码值必须和接口契约一致。
前端只能展示翻译，不能改变码值。
```

如果码值需要展示中文，使用前端字典或后端字典接口转换，不能改变原始码值。

## 3. 布尔型码值

数据库布尔型码值统一使用数字：

```text
1 = 是 / 有效 / 已删除
0 = 否 / 无效 / 未删除
```

字段示例：

| 字段 | 码值 | 含义 |
|---|---|---|
| is_valid | 1 | 有效 |
| is_valid | 0 | 无效 |
| is_deleted | 0 | 未删除 |
| is_deleted | 1 | 已删除 |

后端 Entity 中建议使用：

```text
Integer isValid
Integer isDeleted
```

不建议在 Entity 中使用 Boolean 承接数据库码值。

## 4. 状态型码值

状态型码值优先使用稳定英文小写字符串。

示例：

| 字段 | 码值 | 含义 |
|---|---|---|
| status | draft | 草稿 |
| status | pending | 等待处理 |
| status | generating | 生成中 |
| status | completed | 已完成 |
| status | failed | 失败 |

后端必须使用常量类或枚举集中管理。

不得在代码中散落字符串。

## 5. 后端常量命名

布尔码值建议集中放在：

```text
common/DataFlag.java
```

示例：

```text
VALID = 1
INVALID = 0
DELETED = 1
NOT_DELETED = 0
SYSTEM_USER_ID = 0L
```

业务状态码值建议放在业务常量类：

```text
domain/video/constant/VideoTaskStatus.java
```

示例：

```text
DRAFT = "draft"
PENDING = "pending"
GENERATING = "generating"
COMPLETED = "completed"
FAILED = "failed"
```

## 6. 接口契约要求

接口契约必须记录所有码值：

```text
字段名
码值
中文含义
是否可作为请求入参
是否可作为响应字段
```

如果字段不对前端暴露，也要在数据库结构文档中说明。

## 7. 前端使用规则

前端可以做展示翻译：

```text
draft -> 草稿
pending -> 等待处理
```

但前端不得提交未定义码值。

不得把后端码值改造成另一套：

```text
draft -> 0
completed -> 1
```

除非接口契约明确。

## 8. AI 禁止事项

AI 不得：

- 数据库存 1/0，后端 Entity 用 Boolean 后又暴露 true/false。
- 同一字段同时出现 true/false、1/0、Y/N 多套码值。
- 在 Service 中散落字符串状态。
- 绕过接口契约新增状态码。
- 前端自行发明状态值。

## 9. 一句话原则

```text
数据库有码值，后端有常量，接口有契约，前端只翻译不改码。
```
