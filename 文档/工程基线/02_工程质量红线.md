# 工程质量红线

## 1. 目的

本文件用于补充工程基线，明确 AI 写代码时必须遵守的质量红线。

它关注：框架依赖、工程分层、安全、可维护性、测试、配置、日志和错误处理。

## 2. 框架依赖红线

### 前端允许

```text
Vue 3
Vite
TypeScript
Vue Router
Pinia
ESLint
Prettier
Vitest
Playwright 或 Cypress
```

前端需要审批后才能引入：UI 组件库、图表库、动画库、请求库、工具函数库。

前端禁止 AI 自行引入：React、Angular、Nuxt、大型 UI 套件、低维护频率依赖、不明确来源的依赖。

### 后端允许

```text
Spring Boot
Spring Web
Spring Validation
Spring Data JPA 或 MyBatis（二选一，按项目决定）
Spring Security（需要登录认证时再引入）
Flyway 或 Liquibase（二选一，按项目决定）
```

后端禁止 AI 自行引入：Spring Cloud、注册中心、配置中心、服务网关、消息队列、分布式事务、复杂微服务框架。

## 3. 分层红线

前端分层：

```text
pages       页面级组件
components  可复用组件
services    API 调用
types       类型定义
stores      状态管理
styles      样式和设计 token
utils       通用工具
```

后端分层：

```text
api         Controller，请求响应
service     业务逻辑
repository  数据访问
model       实体或领域对象
dto         请求和响应对象
config      配置
exception   异常和错误响应
```

红线：Controller 不写业务逻辑，不直接访问数据库；Repository 不写业务判断；Service 不处理 HTTP 细节；DTO 不直接复用数据库实体给前端。

## 4. 安全红线

AI 不得提交真实密码、token、密钥，不得把 ngrok 地址写死进代码，不得把数据库端口暴露到公网，不得忽略输入校验，不得直接拼接 SQL，不得返回后端异常堆栈给前端。

必须使用 `.env.example` 说明环境变量，使用统一错误响应，对请求参数和外部输入做合法性检查，CORS 必须明确配置。

## 5. 可维护性红线

AI 不得把所有代码写进一个文件，不得一次任务重构大量无关文件，不得新增无文档说明的工具函数，不得复制粘贴大量相似逻辑，不得用魔法字符串表达业务状态。

必须做到文件职责单一、类型定义清楚、错误处理集中、业务状态使用枚举或常量、代码改动能追溯到任务包。

## 6. 测试、配置和日志红线

前端至少需要构建通过、核心组件或核心流程测试、关键页面状态测试。

后端至少需要 Service 单元测试、API 基础测试、参数校验测试、错误响应测试。

`.env` 不提交，`.env.example` 必须提交。后端不打印敏感信息，不把异常堆栈返回给前端。前端 API 失败必须有错误状态。

## 7. AI 执行红线

AI 每次代码交付必须说明：改了哪些文件、为什么改、对应哪个任务包、如何启动、如何验证、风险点、是否需要补约束框架或工程规范。
