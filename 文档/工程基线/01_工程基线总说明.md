# 工程基线总说明

## 1. 为什么需要工程基线

如果工程完全从 0 开始，就不能直接让 AI 写代码。

必须先定义：

```text
前端怎么组织
后端怎么组织
API 怎么对接
数据库怎么设计
本地环境怎么启动
可选部署目标怎么适配
AI 每次能改什么
```

否则 AI 会自己搭工程，后续很难维护。

## 3. 推荐目标项目结构

这里描述的是“被 AI 生成或维护的目标项目结构”，不是当前方法论仓库自身的目录结构。

```text
project-root/
  docs/
  web-admin/
  h5/
  weapp/
  ios/
  backend/
  infra/
  scripts/
  README.md
```

只保留当前产品版本需要的端。

多个前端端是独立工程，不建立前端共享层。

## 4. 前端边界

前端只负责页面展示、组件拆分、路由、状态管理、API 调用封装、页面状态展示和样式实现。

前端不得自行决定新增后端接口、新增业务规则、新增未定义字段或更换技术栈。

不同前端端之间不得通过共享前端代码保持一致性。

多端一致性只能通过接口契约、产品定义、界面规格、工程规格、代码任务包和检查清单保持。

## 5. 后端边界

后端只负责 API 实现、业务逻辑、数据访问、错误响应、配置读取和测试。

后端不得自行引入 Spring Cloud、拆成微服务、新增未定义业务模块、绕过 API 契约或把敏感信息写入代码。

后端默认提供 `/api/v1` 接口，并通过 `/api/v1/status` 提供健康检查。

## 6. 基础设施边界

基础设施只负责本地软件安装说明、端口规划、nginx 反向代理、ngrok 内网穿透、环境变量样例、本地依赖服务启动和可选部署适配说明。

不得自行引入 Kubernetes、复杂 CI/CD、生产级网关、注册中心或配置中心。

如果选择微信云托管，必须参考：

```text
文档/工程基线/04_微信云托管适配方案.md
```

微信云托管只作为可选部署目标，不改变后端单体基线，不改变多端独立工程原则。

## 7. 一句话原则

```text
从 0 到代码，不是直接让 AI 写；
而是先让 AI 进入一个被工程规范约束好的施工现场。
```
