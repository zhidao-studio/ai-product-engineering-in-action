# 数据库设计规范

## 1. 目的

本文件定义关系型数据库设计规范，用于约束 AI 生成数据库结构、Flyway migration、Entity、Mapper 和接口字段。

目标：

```text
表结构可读
字段可追溯
索引可解释
迁移可回滚
前后端字段一致
AI 不随意发明字段
```

## 2. 适用范围

默认适用于：

```text
MySQL
Spring Boot
MyBatis-Plus
Flyway
```

如果项目使用 PostgreSQL、SQLite、OceanBase、TiDB 等数据库，必须在工程规格中单独说明差异。

## 3. 基本原则

```text
先有业务词汇表，再有表名字段名。
先有接口契约，再有返回字段。
先有数据库结构文档，再写 migration。
所有表结构变更必须通过 Flyway。
每张业务表默认包含 7 个基础要素。
主键 id 统一使用 32 位字符串。
时间字段默认由数据库 trigger 维护。
多表查询默认使用主键 id 关联。
不得直接让 AI 临时拼 SQL 改表。
```

## 4. 表和字段命名

表名使用 snake_case，推荐业务单数名：

```text
video_task
wechat_user
user_role
role_permission
```

字段使用 snake_case：

```text
id
title
status
created_by
created_at
updated_by
updated_at
is_valid
is_deleted
```

前后端映射：

```text
数据库字段：script_summary
Java 字段：scriptSummary
API 字段：scriptSummary
前端字段：scriptSummary
```

禁止：

```text
VideoTask
videoTasks
t_video_task
tbl_video_task
中文字段名
同一字段多种英文名
```

## 5. 每张表 7 要素

每张业务表默认必须包含：

| 字段 | 类型建议 | Java 字段 | 说明 |
|---|---|---|---|
| id | CHAR(32) PRIMARY KEY | id | 32 位字符串主键，禁止自增 BIGINT |
| created_by | CHAR(32) NOT NULL DEFAULT '00000000000000000000000000000000' | createdBy | 创建人 |
| created_at | TIMESTAMP NOT NULL | createdAt | 创建时间 |
| updated_by | CHAR(32) NOT NULL DEFAULT '00000000000000000000000000000000' | updatedBy | 更新人 |
| updated_at | TIMESTAMP NOT NULL | updatedAt | 更新时间 |
| is_valid | TINYINT(1) NOT NULL DEFAULT 1 | isValid | 是否有效 |
| is_deleted | TINYINT(1) NOT NULL DEFAULT 0 | isDeleted | 是否删除 |

说明：

- `id` 是所有业务表的主键，统一为 32 位字符串。
- `created_by`、`updated_by` 记录用户表的 `id`。
- 无登录态或系统任务时，创建人和更新人使用 32 位 0 字符串。
- `created_at`、`updated_at` 默认由数据库 trigger 维护。
- `is_valid` 用于业务有效性控制。
- `is_deleted` 用于逻辑删除。

非业务字典表、关系表、日志表是否裁剪 7 要素，必须在数据库结构文档中说明原因。

## 6. 主键规范

默认主键：

```text
id CHAR(32) PRIMARY KEY
```

Java 侧：

```text
String id
```

主键生成规则：

```text
应用层生成 32 位字符串 ID。
数据库只负责存储，不负责自增生成。
```

推荐 ID 形态：

```text
32 位无横线 UUID
例如：550e8400e29b41d4a716446655440000
```

禁止：

```text
BIGINT AUTO_INCREMENT
Long id
IdType.AUTO
```

## 7. 创建人和更新人

默认字段：

```text
created_by CHAR(32) NOT NULL DEFAULT '00000000000000000000000000000000'
updated_by CHAR(32) NOT NULL DEFAULT '00000000000000000000000000000000'
```

Java 字段：

```text
String createdBy
String updatedBy
```

规则：

- 有登录态时，写入当前用户表的 32 位字符串主键 ID。
- 无登录态或系统任务时，默认使用 `00000000000000000000000000000000`。
- 不使用用户名、手机号、昵称等可变信息作为创建人或更新人字段。
- 不使用数字 `0` 作为系统用户 ID。

## 8. 时间字段和 Trigger

默认字段：

```text
created_at TIMESTAMP NOT NULL
updated_at TIMESTAMP NOT NULL
```

Java 字段：

```text
LocalDateTime createdAt
LocalDateTime updatedAt
```

时间字段默认由数据库 trigger 维护：

- insert 时自动填充 `created_at` 和 `updated_at`。
- update 时自动刷新 `updated_at`。
- 应用代码不得依赖手工赋值作为唯一来源。

MySQL trigger 命名：

```text
trg_表名_bi
trg_表名_bu
```

跨时区项目必须在工程规格中明确：

```text
数据库存 UTC
前端按用户时区展示
```

## 9. 有效和删除码值

默认字段：

```text
is_valid TINYINT(1) NOT NULL DEFAULT 1
is_deleted TINYINT(1) NOT NULL DEFAULT 0
```

Java 字段：

```text
Integer isValid
Integer isDeleted
```

语义：

```text
is_valid = 1   有效
is_valid = 0   无效
is_deleted = 0 未删除
is_deleted = 1 已删除
```

查询业务数据时默认过滤：

```text
is_valid = 1
is_deleted = 0
```

不建议在 Entity 中使用 Boolean 承接数据库码值。

是否启用 MyBatis-Plus 逻辑删除，必须在工程规格中明确。

## 10. 状态字段

状态字段统一使用：

```text
status VARCHAR(50) NOT NULL
```

状态值优先使用稳定英文小写字符串：

```text
draft
pending
generating
completed
failed
```

除非接口契约明确使用数字枚举，否则不得混用：

```text
DRAFT
Draft
草稿
0
1
```

## 11. 字段类型建议

| 场景 | 推荐类型 |
|---|---|
| 主键 | CHAR(32) |
| 用户 ID / 关联 ID | CHAR(32) |
| 短文本 | VARCHAR(100) / VARCHAR(200) |
| 长文本 | TEXT |
| 状态 | VARCHAR(50) |
| 金额 | DECIMAL(18,2) |
| 数量 | INT / BIGINT |
| 比率 | DECIMAL(10,4) |
| 时间 | TIMESTAMP / DATETIME |
| 布尔码值 | TINYINT(1) |
| JSON | JSON，需项目确认 |

不要用 FLOAT / DOUBLE 存金额。

## 12. 非空、默认值和索引

规则：

- 必须字段加 NOT NULL。
- 7 要素字段必须有明确默认值或 trigger 维护策略。
- 状态字段必须 NOT NULL。
- 时间字段必须 NOT NULL。
- 可选字段允许 NULL，但接口契约要说明为空含义。

索引命名：

```text
普通索引：idx_表名_字段名
唯一索引：uk_表名_字段名
```

索引必须服务明确查询场景，不得为了“可能会用”随意加索引。

唯一约束必须来自业务规则，不要对展示名称、标题、备注随意加唯一约束。

## 13. 关联字段和多表查询

多表查询默认使用主键 `id` 关联。

关联字段命名：

```text
业务对象_id
```

示例：

```text
user_id
role_id
video_task_id
wechat_user_id
```

规则：

- 关联字段类型必须与被关联表主键一致：`CHAR(32)`。
- 被关联表必须使用 `id` 作为主键。
- 查询时使用 `xxx_id = xxx.id` 关联。
- 不使用名称、编码、手机号、openid 等非主键字段作为默认关联字段。

如果必须使用业务编码关联，必须在数据库结构文档中说明原因。

## 14. 外键规范

默认不强制使用数据库外键。

如果项目要求强外键，必须在工程规格中说明，并同步 migration 和删除策略。

## 15. Flyway migration 规范

所有表结构变更必须通过 Flyway migration。

命名格式：

```text
V版本号__动作_业务对象.sql
```

示例：

```text
V1__create_video_task.sql
V2__add_status_to_video_task.sql
V3__create_wechat_user.sql
```

禁止修改已经在共享环境执行过的历史 migration。

新增变更必须新增 migration 文件。

如果 migration 中包含 trigger，必须在模板验证阶段确认当前数据库和 Flyway 版本可执行。

## 16. 数据库结构文档

每个项目必须有：

```text
目标项目/docs/05_数据库结构.md
```

每张表至少记录：

```text
表名
表说明
字段清单
7 要素字段
主键生成规则
主键类型
索引
唯一约束
状态枚举
关联字段
和接口字段的映射关系
是否使用 trigger
是否启用逻辑删除
```

## 17. Entity 映射规范

Java Entity 必须和数据库表字段可追溯。

示例：

```text
video_task.id         → VideoTask.id         → String
video_task.title      → VideoTask.title
video_task.created_at → VideoTask.createdAt
video_task.is_valid   → VideoTask.isValid
video_task.is_deleted → VideoTask.isDeleted
```

Entity 不直接返回给前端。

对外返回必须使用 VO。

## 18. 分页查询

列表接口默认支持分页。

请求参数：

```text
pageNo
pageSize
```

默认值：

```text
pageNo = 1
pageSize = 20
```

最大 pageSize 必须限制，例如：

```text
pageSize <= 100
```

第一版样板可先不实现分页，但真实业务列表必须补分页策略。

## 19. AI 禁止事项

AI 不得：

- 未更新数据库结构文档就新增表。
- 未更新接口契约就新增字段。
- 未写 Flyway migration 就改表结构。
- 直接修改已经执行过的 migration。
- 新建业务表却缺少 7 要素字段。
- 使用自增 BIGINT 作为业务表主键。
- 使用 `Long id` 作为业务主键字段。
- 使用中文字段名。
- 混用 camelCase 和 snake_case 数据库字段。
- 使用 FLOAT / DOUBLE 存金额。
- 随意新增同义字段。
- 使用名称、手机号、openid 等非主键字段作为默认表关联字段。
- 把 Entity 直接返回给前端。

## 20. 一句话原则

```text
主键 32 位字符串，每表 7 要素，时间 trigger 管，主键 id 关联，字段全链路可追溯。
```
