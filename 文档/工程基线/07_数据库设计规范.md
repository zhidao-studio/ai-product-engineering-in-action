# 数据库设计规范

## 1. 目的

本文件定义关系型数据库的设计规范，用于约束 AI 生成数据库结构、Flyway migration、Entity、Mapper 和接口字段。

数据库规范的目标是：

```text
表结构可读
字段可追溯
索引可解释
迁移可回滚
前后端字段一致
AI 不随意发明字段
```

## 2. 适用范围

本规范默认适用于：

```text
MySQL
Spring Boot
MyBatis-Plus
Flyway
```

如果项目使用 PostgreSQL、SQLite、OceanBase、TiDB 等数据库，必须在工程规格中单独说明差异。

## 3. 基本原则

```text
先有业务词汇表，再有表名字段名。
先有接口契约，再有返回字段。
先有数据库结构文档，再写 migration。
所有表结构变更必须通过 Flyway。
不得直接让 AI 临时拼 SQL 改表。
```

## 4. 表命名规范

表名使用 snake_case。

推荐使用业务单数名：

```text
video_task
wechat_user
user_role
role_permission
```

不推荐：

```text
VideoTask
videoTasks
t_video_task
tbl_video_task
sys_video_task
```

除非项目已有历史规范，否则不要默认加 `t_`、`tbl_`、`sys_` 前缀。

## 5. 字段命名规范

字段使用 snake_case。

示例：

```text
id
title
topic
script_summary
storyboard_summary
status
created_at
updated_at
```

前后端映射：

```text
数据库字段：script_summary
Java 字段：scriptSummary
API 字段：scriptSummary
前端字段：scriptSummary
```

不得出现同一字段多种英文名：

```text
scriptSummary
scriptDesc
scriptText
scriptContent
```

除非它们在业务词汇表中是不同概念。

## 6. 主键规范

默认主键字段：

```text
id BIGINT PRIMARY KEY
```

Java 侧默认使用：

```text
Long id
```

MyBatis-Plus 可使用：

```text
IdType.ASSIGN_ID
```

如果项目需要数据库自增、自定义雪花 ID、UUID，必须在工程规格中明确。

## 7. 必备字段规范

业务主表默认建议包含：

```text
id
created_at
updated_at
```

如果需要软删除，增加：

```text
deleted
```

如果需要审计，增加：

```text
created_by
updated_by
```

如果需要乐观锁，增加：

```text
version
```

第一版内部 MVP 可只保留：

```text
id
created_at
updated_at
```

## 8. 时间字段规范

数据库字段使用：

```text
created_at TIMESTAMP NOT NULL
updated_at TIMESTAMP NOT NULL
```

Java 使用：

```text
LocalDateTime createdAt
LocalDateTime updatedAt
```

API 返回格式必须在接口契约中说明。

跨时区项目必须在工程规格中明确：

```text
数据库存 UTC
前端按用户时区展示
```

## 9. 状态字段规范

状态字段统一使用：

```text
status VARCHAR(50) NOT NULL
```

状态值使用小写英文或接口契约明确的枚举值。

示例：

```text
draft
pending
generating
completed
failed
```

不得混用：

```text
DRAFT
Draft
草稿
0
1
```

除非接口契约明确使用数字枚举。

## 10. 字段类型建议

常用字段类型：

| 场景 | 推荐类型 |
|---|---|
| 主键 | BIGINT |
| 短文本 | VARCHAR(100) / VARCHAR(200) |
| 长文本 | TEXT |
| 状态 | VARCHAR(50) |
| 金额 | DECIMAL(18,2) |
| 数量 | INT / BIGINT |
| 比率 | DECIMAL(10,4) |
| 时间 | TIMESTAMP / DATETIME |
| 布尔 | TINYINT(1) 或 BOOLEAN |
| JSON | JSON，需项目确认 |

不要用 FLOAT / DOUBLE 存金额。

## 11. 非空和默认值规范

必须字段加 NOT NULL。

可选字段允许 NULL，但接口契约要说明为空含义。

状态字段必须 NOT NULL。

时间字段必须 NOT NULL。

布尔字段建议有默认值：

```text
deleted TINYINT(1) NOT NULL DEFAULT 0
```

## 12. 索引命名规范

普通索引：

```text
idx_表名_字段名
```

唯一索引：

```text
uk_表名_字段名
```

示例：

```text
idx_video_task_status
idx_video_task_created_at
uk_user_username
```

索引必须服务于明确查询场景，不得为了“可能会用”随意加索引。

## 13. 唯一约束规范

唯一约束必须来自业务规则。

示例：

```text
username 唯一
wechat_openid 唯一
业务编码 code 唯一
```

不要对展示名称、标题、备注随意加唯一约束。

## 14. 外键规范

默认不强制使用数据库外键。

可以使用逻辑外键字段，例如：

```text
user_id
role_id
video_task_id
```

如果项目要求强外键，必须在工程规格中说明，并同步迁移脚本和删除策略。

## 15. 软删除规范

如需要软删除，统一使用：

```text
deleted TINYINT(1) NOT NULL DEFAULT 0
```

查询默认过滤：

```text
deleted = 0
```

是否启用 MyBatis-Plus 逻辑删除，必须在工程规格中明确。

第一版 MVP 如果没有恢复、审计、合规需求，可以不启用软删除。

## 16. Flyway migration 规范

所有表结构变更必须通过 Flyway migration。

命名格式：

```text
V版本号__动作_业务对象.sql
```

示例：

```text
V1__create_video_task.sql
V2__add_status_to_video_task.sql
V3__create_wechat_user.sql
```

禁止修改已经在共享环境执行过的历史 migration。

新增变更必须新增 migration 文件。

## 17. 数据库结构文档规范

每个项目必须有：

```text
目标项目/docs/05_数据库结构.md
```

每张表至少记录：

```text
表名
表说明
字段清单
主键
索引
唯一约束
状态枚举
和接口字段的映射关系
是否软删除
是否审计
```

## 18. Entity 映射规范

Java Entity 必须和数据库表字段可追溯。

示例：

```text
video_task.title      → VideoTask.title
video_task.created_at → VideoTask.createdAt
```

Entity 不直接返回给前端。

对外返回必须使用 VO。

## 19. 分页查询规范

列表接口默认支持分页。

请求参数建议：

```text
pageNo
pageSize
```

默认值建议：

```text
pageNo = 1
pageSize = 20
```

最大 pageSize 必须限制，例如：

```text
pageSize <= 100
```

第一版样板可先不实现分页，但真实业务列表必须补分页策略。

## 20. AI 禁止事项

AI 不得：

- 未更新数据库结构文档就新增表。
- 未更新接口契约就新增字段。
- 未写 Flyway migration 就改表结构。
- 直接修改已经执行过的 migration。
- 使用中文字段名。
- 混用 camelCase 和 snake_case 数据库字段。
- 使用 FLOAT / DOUBLE 存金额。
- 随意新增同义字段。
- 把 Entity 直接返回给前端。

## 21. 一句话原则

```text
业务词汇定表名，接口契约定字段，Flyway 管变更，Entity 可追溯，VO 对外暴露。
```
