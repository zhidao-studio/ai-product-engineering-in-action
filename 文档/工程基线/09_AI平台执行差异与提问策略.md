# AI 平台执行差异与提问策略

## 1. 目的

不同 AI 编码平台的默认行为不一致。

有的平台偏向先提问再执行，有的平台偏向直接根据已有上下文执行。

本文件用于约束 AI 平台在文档控制系统中的执行行为，避免因为平台默认风格不同导致流程失控。

## 2. 常见差异

### 偏提问型平台

表现：

```text
先问要不要做
先问是否确认
先问技术选择
先问目录结构
先问是否继续
```

风险：

```text
中断任务包执行
把已经在文档里确定的内容重新抛给使用者
拖慢交付
导致 AI 逃避执行
```

### 偏执行型平台

表现：

```text
根据现有文档直接改代码
较少追问
倾向一次性完成任务
```

风险：

```text
如果文档不完整，可能直接猜
如果任务包边界不清，可能改多
如果检查清单不足，可能遗漏验收
```

## 3. 本系统的默认策略

本系统默认采用：

```text
文档已定义的内容，不再提问。
任务包已定义的范围，直接执行。
只有真正阻塞执行的问题，才允许提问。
```

## 4. 允许提问的情况

只有以下情况允许 AI 提问：

```text
缺少必须输入文件，且无法从上下文推断。
任务包没有说明目标端或目标模块。
两个已确认文档存在直接冲突。
执行会删除用户代码或造成不可逆修改。
需要真实密钥、账号、证书、生产环境地址。
测试或构建需要用户本地环境才能继续。
```

## 5. 不允许提问的情况

以下情况不得提问，必须按文档和任务包执行：

```text
产品定义已经写明。
界面规格已经写明。
原型确认记录已经写明。
工程规格已经写明。
工程模板已经指定。
接口契约已经写明。
数据库结构已经写明。
代码任务包已经写明允许修改范围。
检查清单已经写明验收标准。
```

## 6. 不确定时的处理

如果存在非阻塞不确定项，AI 应该：

```text
采用文档中最保守的默认方案。
在输出中记录假设。
继续完成当前任务。
把不确定项列入风险点或待确认事项。
```

不得因为非阻塞不确定项停止执行。

## 7. 针对偏提问型平台的前置指令

使用偏提问型平台时，在任务开头加入：

```text
不要先问我是否确认。
不要重复询问文档中已经确定的内容。
除非缺少会阻塞执行的必要信息，否则直接执行任务包。
如果存在非阻塞不确定项，先按工程基线默认方案处理，并在结果中列出假设和风险。
```

## 8. 针对偏执行型平台的前置指令

使用偏执行型平台时，在任务开头加入：

```text
严格只执行当前任务包。
不得扩展任务范围。
不得改动任务包未允许修改的文件。
不得从 0 重建工程。
不得绕过工程基线、命名规范、数据库规范和码值规范。
完成后必须按检查清单说明验证结果。
```

## 9. 平台选择建议

如果任务目标是：

```text
修复小问题
执行明确任务包
批量改模板
补齐规范文档
```

优先使用偏执行型平台。

如果任务目标是：

```text
需求不清
设计方向不明
技术路线还没定
产品边界需要讨论
```

可以使用偏提问型平台，但必须先让它输出决策建议，而不是进入编码。

## 10. 一句话原则

```text
讨论阶段可以提问，执行阶段默认动手；文档已定不再问，真阻塞才提问。
```
