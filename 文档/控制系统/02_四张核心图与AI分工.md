# 四张核心图与 AI 分工

本文不是新的方法论。

本文只是把 `00_系统总览.md` 中已经冻结的结构，转成四张可维护的图。

四张图分别回答四个问题：

```text
1. 整个系统是什么？
2. 产品从想法到交付怎么流动？
3. AI 产出如何被门禁拦住？
4. 哪些工作可以交给 AI，哪些必须人确认？
```

## 2. 阶段流程图

回答：产品从想法到交付，必须经过哪些阶段？

```mermaid
stateDiagram-v2
    [*] --> idea
    idea --> product_definition_draft: 输出问题 / 选项 / 产品定义草案
    product_definition_draft --> product_definition_confirmed: 使用者确认产品定义
    product_definition_confirmed --> ui_spec_draft: 生成界面规格
    ui_spec_draft --> prototype_generated: 生成高保真交互预览
    prototype_generated --> prototype_confirmed: 使用者确认原型
    prototype_confirmed --> engineering_spec_confirmed: 生成并确认工程规格
    engineering_spec_confirmed --> contract_confirmed: 确认 OpenAPI / 数据库契约
    contract_confirmed --> task_package_confirmed: 确认代码任务包
    task_package_confirmed --> coding: AI 在模板内编码
    coding --> verification: 执行检查与测试
    verification --> acceptance: 通过门禁后进入验收
    acceptance --> delivered: 使用者验收通过
    delivered --> idea: 新一轮迭代

    verification --> coding: 失败后按修复任务回退
    acceptance --> prototype_generated: 体验不符合预期时回到原型层
    contract_confirmed --> engineering_spec_confirmed: 契约冲突时回退工程规格
```

阶段规则：

```text
AI 不能从 idea 直接跳到完整案例、工程规格、接口契约、数据库结构或代码。
每个阶段都必须明确：允许输出、禁止输出、进入下一阶段条件、是否需要用户确认。
```

## 5. AI 分工表

| 阶段 | 可以主要交给 AI | 必须人确认 | 必须门禁检查 |
|---|---|---|---|
| 想法阶段 | 问题清单、选项、推荐方案、产品定义草案 | 产品目标、目标用户、MVP、不做清单 | 阶段不得越级 |
| 产品定义阶段 | PRD 草案、验收标准草案、边界整理 | 是否值得做、是否进入下一阶段 | 文档完整性 |
| 界面规格阶段 | 页面清单、状态清单、交互说明 | 页面范围、核心流程、字段含义 | 不得生成正式代码 |
| 原型阶段 | 高保真 HTML 原型、模拟数据、状态展示 | 视觉与交互是否符合预期 | 不得写后端和数据库 |
| 工程规格阶段 | 工程规格草案、模块拆分、接口建议 | 采用哪些端、功能边界、风险接受 | 是否引用工程基线 |
| 契约阶段 | OpenAPI、database-schema、依赖白名单草案 | 业务字段、状态、权限、关键规则 | OpenAPI / Schema / 白名单检查 |
| 任务包阶段 | 任务拆分、允许修改范围、验收命令 | 任务顺序、范围是否过大 | 任务包完整性 |
| 编码阶段 | 模板内代码实现、测试补充、修改说明 | 是否接受实现结果 | 阶段、边界、契约、依赖门禁 |
| 验证阶段 | 测试执行、失败分析、人话报告草案 | 选择修复方案、是否验收 | 构建、测试、安全、契约检查 |
| 迭代阶段 | 变更影响分析、回归风险提示 | 是否继续迭代、是否回滚 | 回归门禁和变更记录 |

## 6. 分工原则

```text
人做意图判断。
AI 做结构化生成和受控执行。
门禁做强制阻断。
反馈层把技术失败翻译成人话选项。
```

不能交给 AI 独立决定的事情：

```text
产品是否值得做
核心用户是谁
MVP 是否收敛
关键业务规则如何取舍
安全风险是否接受
是否上线
是否回滚
```

适合交给 AI 的事情：

```text
把人话整理成结构化问题
把选项写清楚
把确认后的内容变成契约和任务包
在模板内按任务包编码
执行检查命令
把失败原因翻译成可选择的修复方案
```

必须由门禁拦截的事情：

```text
越级生成
越界修改
缺少契约
修改保护文件
新增白名单外依赖
数据库结构不符合基线
接口不符合 OpenAPI
```

## 7. 使用者模式

如果你是这套控制系统的使用者，你面对的不是命令行，而是执行 AI。

使用者不应该直接执行：

```text
git pull
npm install
mvn test
bash scripts/check-ai-guard.sh
```

这些属于执行 AI、工程 AI、验证 AI 或自动门禁系统的工作。

使用者只需要做三类动作：

```text
1. 说清楚目标。
2. 在 AI 给出的选项中做选择。
3. 根据人话报告确认、拒绝、回滚或继续。
```

### 7.1 使用者应该怎么发起任务

使用者应该对执行 AI 说：

```text
请按本仓库控制系统执行下一步。

你必须先读取：
1. 文档/控制系统/00_系统总览.md
2. 文档/控制系统/02_四张核心图与AI分工.md
3. 文档/控制系统/01_阶段状态机.md
4. 目标项目/AI_CONTEXT_PACK/

然后告诉我：
- 当前阶段是什么
- 你准备做什么
- 需要我确认什么
- 哪些事情你会自己执行
- 哪些事情必须门禁检查

如果需要执行命令，由你执行。
如果命令失败，请把失败原因翻译成人话，并给我 2-3 个选择。
不要让我直接操作命令行。
```

### 7.2 执行 AI 应该怎么工作

执行 AI 应该自己完成：

```text
读取仓库文档
判断当前阶段
读取 AI_CONTEXT_PACK
生成或修改目标文件
执行构建 / 测试 / 门禁命令
整理失败原因
输出人话报告
等待使用者确认
```

执行 AI 不应该把命令丢给使用者执行，除非：

```text
当前 AI 环境没有终端权限
当前 AI 无法访问本地仓库
当前命令涉及账号登录、密钥、付费、系统权限或外部工具授权
```

遇到这些情况，执行 AI 应该明确说明：

```text
我现在无法直接执行该操作，原因是……
这一步属于环境权限问题，不是产品决策问题。
你可以选择：
A. 换到有终端权限的执行 AI 中运行
B. 授权当前 AI 执行
C. 暂时跳过该验证，仅记录为未验证风险
```

### 7.3 使用者不应该被要求判断什么

使用者不应该被要求判断：

```text
构建命令应该怎么写
依赖怎么安装
脚本怎么运行
Git hook 怎么配置
CI YAML 怎么写
Maven / npm / Xcode 报错怎么修
```

这些问题必须由执行 AI 先分析、先修复、先翻译。

使用者只判断：

```text
这个产品方向是否继续
这个原型是否接受
这个业务规则是否确认
这个风险是否接受
这个修复方案选哪个
这个结果是否验收
```

### 7.4 正确的人机协作关系

```text
使用者 = 产品负责人 / 决策者
执行 AI = 项目经理 + 工程执行者 + 验证执行者
门禁脚本 = 机器质检员
人话报告 = 决策界面
```

所以正确路径不是：

```text
使用者执行命令 → 把结果贴给 AI → AI 分析
```

而是：

```text
使用者提出目标 → 执行 AI 执行命令和验证 → AI 输出人话报告 → 使用者做决策
```

## 8. 开发者 / 维护者模式

开发者视角不是产品使用视角。

开发者 / 维护者面对的是控制系统本身，主要职责是让这套系统：

```text
模板能跑
门禁能拦
契约能校验
反馈能读懂
文档不失控
```

开发者可以直接操作命令行，因为他的工作对象不是某个产品需求，而是控制系统和工程基线本身。

### 8.1 开发者负责什么

开发者负责：

```text
维护工程模板
验证模板能否构建和运行
修复脚本和 CI
增强 OpenAPI / Database Schema 校验
维护依赖白名单
维护工程基线、命名规范、数据库规范、码值规范
处理模板升级和兼容性
处理控制系统自身的回归测试
```

开发者可以执行：

```text
git pull
npm install
npm run build
mvn test
mvn package
bash scripts/check-ai-guard.sh <目标项目目录>
bash scripts/run-ai-guard-all.sh
bash scripts/install-ai-guard-hook.sh
```

这些命令不应该要求普通使用者执行，但可以要求开发者、执行 AI、工程 AI 或 CI 执行。

### 8.2 开发者不负责什么

开发者不应该替使用者决定：

```text
产品是否值得做
目标用户是谁
MVP 是否继续收缩
原型体验是否接受
业务规则如何取舍
风险是否业务上可接受
是否上线
是否回滚
```

这些属于使用者 / 产品负责人。

开发者可以给出技术影响和风险解释，但不能替业务做最终决策。

### 8.3 开发者面对 AI 的方式

开发者可以把 AI 当成工程助手，用于：

```text
修复模板编译问题
补充单元测试
增强门禁脚本
分析 CI 失败原因
整理验证记录
生成脚本变更说明
比较规范和实现是否一致
```

但开发者使用 AI 时，也必须遵守冻结结构：

```text
不新增另一套总览
不新增重复规范
不绕过阶段状态机
不擅自扩大模板技术栈
不为了修一个问题重建整个工程
```

### 8.4 双角色边界表

| 事项 | 使用者 / 产品负责人 | 开发者 / 维护者 | 执行 AI / Agent | 门禁 / CI |
|---|---|---|---|---|
| 提出产品目标 | 负责 | 不负责 | 辅助整理 | 不参与 |
| 选择目标用户 | 负责 | 不负责 | 提供选项 | 不参与 |
| 确认 MVP 范围 | 负责 | 提供技术影响 | 辅助拆解 | 检查是否越级 |
| 确认原型 | 负责 | 可解释技术限制 | 生成原型 | 不参与 |
| 确认接口字段含义 | 负责业务语义 | 负责技术表达 | 生成契约草案 | 检查契约存在性 |
| 维护工程模板 | 不负责 | 负责 | 辅助实现 | 检查保护路径 |
| 执行构建命令 | 不负责 | 负责 | 可代执行 | 可自动执行 |
| 修复编译失败 | 不负责 | 负责 | 辅助修复 | 反馈失败 |
| 增强门禁脚本 | 不负责 | 负责 | 辅助实现 | 自身被验证 |
| 判断是否验收 | 负责 | 提供技术结论 | 生成人话报告 | 提供检查结果 |
| 判断是否上线 | 负责 | 提供上线风险 | 汇总风险 | 阻断不合规项 |
| 判断是否回滚 | 负责 | 提供回滚方案 | 辅助生成方案 | 提供失败证据 |

### 8.5 最关键的边界

```text
使用者不碰命令行。
开发者可以碰命令行。
执行 AI 应尽量替使用者执行命令。
门禁负责把不能通过的结果拦住。
```

所以同一个动作，在不同角色下含义不同：

```text
npm run build
```

对使用者来说：

```text
这是执行 AI 或开发者应该完成的验证。
```

对开发者来说：

```text
这是维护工程模板和确认可交付性的基本动作。
```

对执行 AI 来说：

```text
这是验证任务的一部分，执行后必须翻译结果。
```

对门禁 / CI 来说：

```text
这是自动阻断不可交付结果的依据。
```
