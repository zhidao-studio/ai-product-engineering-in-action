# 控制系统总览

## 0. 一句话定位

本仓库不是提示词集合，也不是单纯工程模板集合。

它当前定位为：

```text
约束 AI 作为生产力的阶段受控产品交付系统。
```

它要解决的不是“AI 能不能写代码”，而是：

```text
人如何把 AI 这种不稳定的生产力，放进可理解、可检查、可回退的产品交付路径里。
```

## 1. 总：AI 作为生产力的使用路径

AI 不是产品负责人，也不是自由工程师。

AI 在本系统里的角色是：

```text
受控执行者。
```

人类在本系统里的角色是：

```text
产品意图和关键决策的负责人。
```

控制系统在中间承担：

```text
把人的意图结构化，
把结构化意图交给 AI 执行，
把 AI 的产出放进工程边界，
把工程结果翻译回人能决策的语言。
```

总路径如下：

```text
人说出产品想法
↓
系统判断当前阶段
↓
系统把想法转成结构化问题、选项和文档
↓
人确认产品方向和界面预期
↓
系统冻结阶段性契约
↓
AI 在工程模板和任务包内执行
↓
机器门禁检查是否越界
↓
失败时翻译成人话选项
↓
人做产品决策
↓
系统进入下一轮交付
```

## 2. 分：四条主线

本仓库所有内容都必须归入以下四条主线。

### 2.1 使用者主线

使用者不需要理解代码。

使用者需要做的是：

```text
说清楚想法
选择方案
确认原型
确认关键业务规则
验收结果
决定是否继续迭代
```

使用者不应该被要求判断：

```text
ORM 选型
数据库索引细节
依赖版本
工程目录结构
CI/CD 配置
事务隔离级别
```

技术细节由工程基线和模板固化。

技术影响必须被系统翻译成产品语言。

### 2.2 AI 主线

AI 每次工作前必须先判断：

```text
当前阶段是什么
当前允许输出什么
当前禁止输出什么
当前允许修改哪些文件
当前任务包只要求做什么
```

AI 的标准执行顺序是：

```text
读取控制系统总览
读取阶段状态机
读取 AI Context Pack
判断当前阶段
判断允许动作
执行当前阶段任务
输出修改说明、验证结果和风险
```

AI 禁止：

```text
跳过阶段直接生成完整项目
跳过原型确认直接编码
未确认接口契约前写接口
未确认数据库结构前写 migration
擅自修改工程模板和保护文件
擅自新增依赖
从 0 重建工程
```

### 2.3 工程主线

工程不是 AI 临时搭出来的。

工程由模板固定边界：

```text
Web 后台模板工程
H5 模板工程
微信小程序模板工程
iOS 模板工程
Spring Boot 后端模板工程
```

工程交付路径：

```text
确认产品定义
↓
确认界面规格
↓
生成高保真交互预览
↓
使用者确认原型
↓
确认工程规格
↓
确认接口契约和数据库结构
↓
确认代码任务包
↓
复制对应工程模板
↓
AI 在模板内编码
↓
测试和检查清单验证
```

工程基线负责固定：

```text
技术栈
目录结构
命名规范
数据库规范
码值规范
多端关系
模板验证状态
```

### 2.4 治理主线

治理不是靠 AI 自觉。

治理分三步逐步变硬：

```text
第一步：文档约束
第二步：结构化契约
第三步：机器门禁
```

当前状态：

```text
文档约束已建立。
结构化契约已开始。
最小机器门禁已实现。
深度契约校验待实现。
```

最终目标：

```text
AI 可以犯错，但错误必须发生在低成本阶段；
AI 可以执行，但执行必须在边界内；
AI 可以失败，但失败必须被翻译成人能决策的选项。
```

## 3. 总分结构图

```text
总目标：约束 AI 作为生产力
│
├─ 使用者主线：想法 → 选择 → 原型确认 → 验收
│
├─ AI 主线：读上下文 → 判阶段 → 判边界 → 执行任务
│
├─ 工程主线：文档 → 原型 → 契约 → 模板 → 编码 → 验证
│
└─ 治理主线：文档约束 → 结构化契约 → 机器门禁 → 人话反馈
```

## 5. AI 的使用方式

### 5.1 规划阶段

适用场景：只有想法，还没有产品定义。

AI 应输出：

```text
当前阶段判断
缺失信息
问题清单
每个问题的选项
推荐方案
下一步 ToDo
```

AI 不得输出：

```text
完整案例目录
工程规格
接口契约
数据库结构
代码任务包
工程代码
```

### 5.2 原型阶段

适用场景：产品定义和界面规格已经确认。

AI 应输出：

```text
高保真交互预览
模拟数据
页面状态
原型确认清单
```

AI 不得输出：

```text
正式工程代码
后端接口实现
数据库 migration
```

### 5.3 工程阶段

适用场景：原型已确认，工程规格、接口契约、数据库结构、代码任务包已确认。

AI 应输出：

```text
受限代码修改
修改说明
验证命令
验证结果
风险和待确认事项
```

AI 不得输出：

```text
扩展任务范围
修改保护文件
擅自新增依赖
重建工程结构
```

### 5.4 验证阶段

适用场景：代码已经生成，需要检查是否可交付。

AI 应输出：

```text
测试结果
检查清单结果
失败原因
人话报告
修复选项
```

AI 不得输出：

```text
隐藏失败
忽略失败
连续盲目重试
未经确认修改契约或数据库
```

## 6. 门禁脚本使用方式

在仓库根目录执行：

```bash
bash scripts/check-ai-guard.sh <目标项目目录>
```

示例：

```bash
bash scripts/check-ai-guard.sh 案例/03_钓鱼爱好者移动端平台案例
```

总入口会依次执行：

```text
check-ai-context-pack.sh
check-ai-stage.sh
check-ai-boundary.sh
```

当前能检查：

```text
目标项目是否具备 AI Context Pack
当前阶段是否出现越级产物
Git diff 是否修改保护路径
```

当前不能替代：

```text
OpenAPI 契约校验
数据库 schema 深度校验
单元测试
端到端测试
安全扫描
真实构建验证
```

## 7. 什么不要再加

为了避免仓库失控，后续默认不要再新增以下类型文件：

```text
新的方法论总论
新的原则大全
新的同类命名规范
新的同类数据库规范
新的同类码值规范
新的重复工程模板说明
新的重复检查清单说明
新的平台好坏评价文档
```

如果确实需要新增文件，必须先回答：

```text
它属于使用者主线、AI 主线、工程主线、治理主线中的哪一条？
它属于文档层、契约层、门禁层、反馈层中的哪一层？
它是否能合并到已有文件？
它是否有执行价值？
它是否会成为新的唯一入口？
```

如果答案不清楚，不新增文件，优先修改已有入口文件。

## 8. 下一步只做什么

当前最有价值的下一步不是继续补文档，而是把治理主线从“最小门禁”推进到“深度契约校验”。

优先级：

```text
1. OpenAPI 契约校验
2. 数据库 schema / DDL 7 要素校验
3. 依赖白名单校验
4. 验证失败人话报告自动生成
```

```text
自动校验 OpenAPI
自动校验数据库 schema
自动安全扫描
自动生成人话报告

```

所以当前系统状态是：

```text
控制协议已建立。
最小机器门禁已实现。
深度契约校验待实现。
```
