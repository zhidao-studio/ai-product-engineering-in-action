# 控制系统总览

## 1. 本仓库现在是什么

本仓库不是普通提示词集合，也不是单纯工程模板集合。

它当前定位为：

```text
阶段受控的 AI 产品交付协议。
```

它要解决的问题是：

```text
非技术用户如何在不直接理解代码细节的情况下，控制 AI 把产品想法推进到可验收的软件交付。
```

## 2. 当前系统只分四层

为了避免文档越加越多，后续所有内容必须归入以下四层之一。

```text
文档层       表达人的意图和已确认决策
契约层       把关键约束结构化，减少 AI 自由解释
门禁层       用脚本或检查阻止 AI 越界
反馈层       把技术结果翻译成人能决策的语言
```

## 3. 四层对应内容

| 层级 | 作用 | 当前已有内容 | 后续允许增强什么 |
|---|---|---|---|
| 文档层 | 给人和 AI 说明目标、边界、流程 | README、AGENTS、产品定义、界面规格、工程规格、方法论、工程基线 | 只补关键入口和缺失说明，不再横向堆同类规范 |
| 契约层 | 把阶段、产品、接口、数据库、权限、任务变成结构化输入 | 阶段状态机、AI Context Pack、产品蓝图、允许修改范围 | 逐步补 OpenAPI、database-schema、task-package.yaml |
| 门禁层 | 自动检查 AI 是否越界 | 目前只有规则，还没有脚本 | 补 scripts/check-ai-stage.sh、check-ai-boundary.sh、check-ai-context-pack.sh |
| 反馈层 | 把验证失败翻译成可选择方案 | 验证失败人话报告样板 | 后续接入测试失败、构建失败、契约失败报告 |

## 4. 当前最重要的主链路

```text
产品想法
↓
判断当前阶段
↓
读取 AI Context Pack
↓
生成当前阶段允许的产物
↓
用户确认
↓
进入下一阶段
↓
复制对应工程模板
↓
按代码任务包执行
↓
机器门禁检查
↓
人话报告和使用者验收
```

## 6. 什么不要再加

为了避免仓库失控，后续默认不要再新增以下类型文件：

```text
新的方法论总论
新的原则大全
新的同类命名规范
新的同类数据库规范
新的同类码值规范
新的重复工程模板说明
新的重复检查清单说明
```

如果确实需要新增文件，必须先回答：

```text
它属于四层中的哪一层？
它是否能替代或合并到已有文件？
它是否会变成新的唯一入口？
它是否有执行价值？
```

如果答案不清楚，不新增文件，优先修改已有入口文件。

## 7. 下一步只做什么

当前最有价值的下一步不是继续补文档，而是补最小门禁脚本：

```text
scripts/check-ai-context-pack.sh
scripts/check-ai-stage.sh
scripts/check-ai-boundary.sh
```

它们分别负责：

```text
检查目标项目是否具备 AI Context Pack
检查当前阶段是否出现越级产物
检查 Git diff 是否修改保护路径
```

## 8. 当前未完成边界

当前还没有真正做到：

```text
自动阻止 AI 越级
自动阻止 AI 修改保护文件
自动校验 OpenAPI
自动校验数据库 schema
自动生成人话报告
自动回滚
```

所以当前系统状态是：

```text
控制协议已建立。
机器门禁待实现。
```

```text
文档表达意图，契约收束歧义，门禁阻止越界，报告帮助用户决策。
```
