# 控制系统总览

## 0. 结构冻结声明

从本文件开始，本仓库的整体结构先冻结。

后续不再随意横向新增方法论、规范、目录或同类说明文件。

后续只允许两类动作：

```text
1. 把本文件中已经定义的某个节点做实。
2. 把某个软约束升级成结构化契约或可执行门禁。
```

如果新增内容不能挂到本文件的总图、四条主线、四层架构或目标项目结构中，不新增。

本文件是当前控制系统唯一总图。

## 1. 一句话定位

本仓库不是提示词集合，也不是单纯工程模板集合。

它当前定位为：

```text
约束 AI 作为生产力的阶段受控产品交付系统。
```

它要解决的不是“AI 能不能写代码”，而是：

```text
人如何把 AI 这种不稳定的生产力，放进可理解、可检查、可回退的产品交付路径里。
```

一句话：

```text
人负责意图和决策；
AI 负责受控执行；
系统负责阶段、契约、边界和反馈。
```

### 3.1 使用者主线

使用者不需要理解代码。

使用者需要做的是：

```text
说清楚想法
选择方案
确认原型
确认关键业务规则
验收结果
决定是否继续迭代
```

使用者不应该被要求判断：

```text
ORM 选型
数据库索引细节
依赖版本
工程目录结构
CI/CD 配置
事务隔离级别
```

技术细节由工程基线和模板固化。

技术影响必须被系统翻译成产品语言。

### 3.2 AI 主线

AI 每次工作前必须先判断：

```text
当前阶段是什么
当前允许输出什么
当前禁止输出什么
当前允许修改哪些文件
当前任务包只要求做什么
```

AI 的标准执行顺序是：

```text
读取控制系统总览
读取阶段状态机
读取 AI Context Pack
判断当前阶段
判断允许动作
执行当前阶段任务
输出修改说明、验证结果和风险
```

AI 禁止：

```text
跳过阶段直接生成完整项目
跳过原型确认直接编码
未确认接口契约前写接口
未确认数据库结构前写 migration
擅自修改工程模板和保护文件
擅自新增依赖
从 0 重建工程
```

### 3.3 工程主线

工程不是 AI 临时搭出来的。

工程由模板固定边界：

```text
Web 后台模板工程
H5 模板工程
微信小程序模板工程
iOS 模板工程
Spring Boot 后端模板工程
```

工程交付路径：

```text
确认产品定义
↓
确认界面规格
↓
生成高保真交互预览
↓
使用者确认原型
↓
确认工程规格
↓
确认接口契约和数据库结构
↓
确认代码任务包
↓
复制对应工程模板
↓
AI 在模板内编码
↓
测试和检查清单验证
```

工程基线负责固定：

```text
技术栈
目录结构
命名规范
数据库规范
码值规范
多端关系
模板验证状态
```

### 3.4 治理主线

治理不是靠 AI 自觉。

治理分三步逐步变硬：

```text
第一步：文档约束
第二步：结构化契约
第三步：机器门禁
```

当前状态：

```text
文档约束已建立。
结构化契约已开始。
最小机器门禁已实现。
最小深度契约校验已实现。
专业级契约校验待实现。
```

最终目标：

```text
AI 可以犯错，但错误必须发生在低成本阶段；
AI 可以执行，但执行必须在边界内；
AI 可以失败，但失败必须被翻译成人能决策的选项。
```

## 4. 四层架构

```text
┌──────────────────────────────────────────────────────────────┐
│  第 4 层：反馈层                                              │
│  作用：把技术结果翻译成人能决策的语言                         │
│  例子：验证失败人话报告、修复选项、风险说明                   │
└──────────────────────────────────────────────────────────────┘
                              ↑
┌──────────────────────────────────────────────────────────────┐
│  第 3 层：门禁层                                              │
│  作用：用脚本或 CI 拦住 AI 越级、越界、契约缺失                │
│  例子：check-ai-guard、stage、boundary、openapi、schema        │
└──────────────────────────────────────────────────────────────┘
                              ↑
┌──────────────────────────────────────────────────────────────┐
│  第 2 层：契约层                                              │
│  作用：把关键约束结构化，减少 AI 自由解释                     │
│  例子：AI_CONTEXT_PACK、OpenAPI、database-schema、白名单       │
└──────────────────────────────────────────────────────────────┘
                              ↑
┌──────────────────────────────────────────────────────────────┐
│  第 1 层：文档层                                              │
│  作用：表达人的意图、流程、规则、已确认决策                   │
│  例子：产品定义、界面规格、工程规格、工程基线、检查清单       │
└──────────────────────────────────────────────────────────────┘
```

层级关系：

```text
文档不是终点；
文档要逐步变成契约；
契约要被门禁检查；
门禁结果要翻译成人话。
```

## 10. AI 的使用方式

### 10.1 规划阶段

适用场景：只有想法，还没有产品定义。

AI 应输出：

```text
当前阶段判断
缺失信息
问题清单
每个问题的选项
推荐方案
下一步 ToDo
```

AI 不得输出：

```text
完整案例目录
工程规格
接口契约
数据库结构
代码任务包
工程代码
```

### 10.2 原型阶段

适用场景：产品定义和界面规格已经确认。

AI 应输出：

```text
高保真交互预览
模拟数据
页面状态
原型确认清单
```

AI 不得输出：

```text
正式工程代码
后端接口实现
数据库 migration
```

### 10.3 工程阶段

适用场景：原型已确认，工程规格、接口契约、数据库结构、代码任务包已确认。

AI 应输出：

```text
受限代码修改
修改说明
验证命令
验证结果
风险和待确认事项
```

AI 不得输出：

```text
扩展任务范围
修改保护文件
擅自新增依赖
重建工程结构
```

### 10.4 验证阶段

适用场景：代码已经生成，需要检查是否可交付。

AI 应输出：

```text
测试结果
检查清单结果
失败原因
人话报告
修复选项
```

AI 不得输出：

```text
隐藏失败
忽略失败
连续盲目重试
未经确认修改契约或数据库
```

## 12. 以后只允许做什么

后续不再“想到一个问题就加一个文件”。

后续只允许两类动作：

```text
1. 把现有图里的某个节点做实。
2. 把某个软约束变成可执行检查。
```

也就是说，后续重点是：

```text
契约更准
门禁更硬
反馈更像人话
工程模板验证更真实
```

默认不要再新增：

```text
新的方法论总论
新的原则大全
新的同类命名规范
新的同类数据库规范
新的同类码值规范
新的重复工程模板说明
新的重复检查清单说明
新的平台好坏评价文档
```

如果确实需要新增文件，必须先回答：

```text
它属于使用者主线、AI 主线、工程主线、治理主线中的哪一条？
它属于文档层、契约层、门禁层、反馈层中的哪一层？
它是否能合并到已有文件？
它是否有执行价值？
它是否会成为新的唯一入口？
```

如果答案不清楚，不新增文件，优先修改已有入口文件。
