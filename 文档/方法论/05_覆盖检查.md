# 覆盖检查

## 1. 定义

覆盖检查用于判断约束框架是否覆盖了 AI 交付中的主要决策点。

核心问题：

```text
AI 可能产出的每个维度，是否都有对应约束？
```

没有对应约束的维度，需要补充模板或检查清单。

## 2. 覆盖检查表

| AI 产出维度 | 对应约束 | 常见缺口 |
|---|---|---|
| 是否新增功能 | PRD / 不做清单 | AI 自行扩功能 |
| 页面数量 | 页面规格 / 设计规格 | 页面越做越多 |
| 页面视觉 | 设计规格 | 风格不一致 |
| 颜色 | 配色规范 | AI 自行配色 |
| 组件样式 | 组件样式规范 | 按钮、卡片、表单各写各的 |
| 组件库选择 | 工程规格 / 技术边界 | AI 自行引入 UI 库 |
| 响应式断点 | 布局规范 / 工程规格 | 手机和桌面表现不一致 |
| 动画和过渡行为 | 设计规格 | AI 自行加动效或无动效 |
| 错误提示文案 | 体验验收 / 设计规格 | 文案生硬或不可理解 |
| 表单校验规则 | 工程规格 / 体验验收 | 输入错误无法判断 |
| 字段名称 | 数据规范 | 字段名不统一 |
| 字段类型 | 数据规范 | 前后端类型不一致 |
| 数据存储方式 | 工程规格 | AI 自行决定本地存储、API 或数据库 |
| 技术选择 | 工程规格 / 工程基线 | AI 自行换框架 |
| 接口路径 | API 规格 | 前后端路径不一致 |
| 接口响应字段 | API 规格 | 响应结构不一致 |
| 修改范围 | 任务执行约束 | AI 一次改太多 |
| 通过标准 | 验证约束 | AI 自己说完成 |
| 用户核心流程 | 体验验收约束 | 能跑但用户不会用 |
| 上线检查 | 交付迭代约束 | 未检查就交付 |

## 3. 合格标准

- 主要产出维度都有约束。
- 每个任务都有允许范围和禁止范围。
- 每个功能都有可检查标准。
- 每个视觉和交互决策都有设计依据。
- 每个字段和接口都有工程规格依据。
- 非技术人能根据文档判断是否通过。
