# 移动端借鉴说明

## 1. 目的

当前工程模板优先覆盖 Web 后台，但真实生产力产品往往还需要移动端入口。

移动端不应直接套用后台模板，而应按端类型分别设计：

```text
H5
微信小程序
iOS
```

## 2. H5 借鉴来源

主要参考：

```text
Vant
```

借鉴内容：

- 移动端组件体系
- 表单、列表、弹窗、Toast、Loading 等通用交互
- lint / test / build 命令体系
- 轻量化移动端页面组织方式

不直接搬入：

- 组件库源码
- 组件文档站点
- 复杂构建体系

H5 模板建议方向：

```text
Vue 3 + Vite + TypeScript + Vant
```

适用场景：

- 移动端管理页
- 活动页
- 表单采集页
- 微信内 H5 页面
- 移动端生产力工具入口

## 3. 微信小程序借鉴来源

主要参考：

```text
Vant Weapp
Taro
```

Vant Weapp 用于参考小程序 UI 组件体系。

Taro 用于参考多端工程组织和编译命令体系。

借鉴内容：

- 小程序组件组织
- 页面和组件目录结构
- 小程序构建、上传、测试命令
- 跨端适配边界

不直接搬入：

- 完整跨端平台能力
- 多端复杂编译配置
- 业务组件和示例页面

小程序模板建议方向：

```text
原生微信小程序 + TypeScript + Vant Weapp
```

或在需要多端时使用：

```text
Taro + TypeScript
```

第一版建议优先原生微信小程序，不要一开始就引入复杂跨端。

## 4. iOS 借鉴来源

主要参考：

```text
SwiftUI 官方应用结构
The Composable Architecture
Kingfisher
```

借鉴内容：

- Swift Package Manager 依赖管理
- SwiftUI 页面组织
- Feature / State / Action / Reducer 的状态管理思路
- 可测试的业务状态和依赖注入
- 图片加载和缓存作为可选能力

不直接搬入：

- 大型架构框架全部能力
- 复杂宏和高级状态管理
- 过多第三方依赖

iOS 模板建议方向：

```text
SwiftUI + MVVM 起步
复杂状态再引入 TCA 思路
```

第一版不建议直接强绑定完整 TCA。

## 5. 移动端模板推进顺序

推荐顺序：

```text
1. H5 模板工程
2. 微信小程序模板工程
3. iOS 模板工程
```

原因：

- H5 最容易和现有后端接口复用
- 小程序适合微信生态和私域场景
- iOS 适合原生体验，但工程和发布门槛更高

## 6. 统一原则

所有移动端模板都必须遵守：

```text
先写产品和界面约束
再复制端模板工程
再写代码任务包
最后让 AI 在模板内开发
```

移动端不得绕过：

- 统一 API 封装
- 统一错误处理
- 页面状态规范
- 设计变量或主题配置
- 测试和构建命令
