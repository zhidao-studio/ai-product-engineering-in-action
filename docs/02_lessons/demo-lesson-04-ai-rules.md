# 第 4 课手把手演示：写 AI 执行规则

## 本节目标

为 AI 编码工具写一份明确的执行规则，避免它乱改工程。

## 起始输入

你已经有了：

```text
项目总说明
产品需求文档
页面清单
用户流程
```

现在要写：

```text
AGENTS.md
```

## 第一步：让 AI 生成执行规则初稿

使用提示词：

```text
请根据下面的项目说明，帮我生成一份 AGENTS.md。

要求：
1. 面向 AI 编码工具使用，例如 Codex、Cursor、Windsurf、Kimi Coding、Trae。
2. 规则要具体，不要空泛。
3. 明确第一版只做什么。
4. 明确禁止做什么。
5. 明确每次只做一个任务。
6. 明确完成后必须输出修改文件、验证方式和风险点。
7. 输出 Markdown。

项目说明：
【粘贴项目总说明、页面清单、用户流程】
```

## 第二步：AI 应该输出什么

合格输出示例：

```markdown
# 本案例 AI 执行规则

## 1. 项目范围

第一版只做钓鱼露营地点收藏小程序。

页面包括：

- 首页
- 详情页
- 收藏页
- 设置页
- 关于页

## 2. 禁止事项

1. 不接入登录。
2. 不接入后端。
3. 不接入支付。
4. 不接入地图。
5. 不接入云开发。
6. 不引入第三方依赖。
7. 不修改无关文件。
8. 不大范围重构。

## 3. 数据规则

第一版只使用 mock 数据和本地存储。

## 4. 任务规则

每次只完成一个 TASK。

## 5. 完成后必须输出

- 修改文件列表
- 实现说明
- 如何验证
- 风险点
- 后续建议
```

## 第三步：检查 AI 输出是否合格

重点检查：

- 是否明确“不做登录、后端、支付、地图”？
- 是否明确“不引入第三方依赖”？
- 是否明确“每次只完成一个 TASK”？
- 是否明确“完成后输出修改文件和风险点”？
- 是否语言足够具体？
- 是否适合放到仓库根目录？

## 第四步：不合格时怎么追问

如果 AI 写得太空泛：

```text
这份规则太空泛，请改成 AI 编码工具能执行的硬性规则。

要求：
1. 必须包含禁止事项。
2. 必须包含允许修改范围原则。
3. 必须包含完成后输出格式。
4. 每条规则都要可检查。
```

如果 AI 没有限制第一版范围：

```text
请补充第一版范围限制：
1. 只做 5 个页面。
2. 只用 mock 数据。
3. 只用本地存储。
4. 不接入登录、后端、地图、支付、云开发。
```

## 第五步：合格产物

最终得到：

```text
AGENTS.md
examples/01_wechat_mini_program_favorites/AGENTS.md
```

## 本节完成标准

- AI 执行规则能约束编码工具。
- 禁止事项明确。
- 完成后输出格式明确。
- 学员能看懂每条规则是什么意思。
- 可以进入第 5 课拆任务和写代码。
