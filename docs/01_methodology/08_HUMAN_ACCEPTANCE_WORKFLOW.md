# 模拟用户验收机制

## 1. 为什么需要模拟用户验收

产品最终交付给人，不是交付给 Agent。

代码能跑，不代表用户能用；功能完成，不代表需求实现正确。

所以在最终交付前，需要增加一个步骤：

```text
模拟用户验收
```

它用于模拟真实用户的使用路径、理解方式和操作逻辑，检查交付内容是否准确。

## 2. 验收位置

完整流程应该是：

```text
产品验证
↓
项目总说明
↓
产品需求文档
↓
页面清单
↓
用户流程
↓
数据规范
↓
技术结构
↓
任务清单
↓
Agent 按任务实现
↓
文档到代码映射审查
↓
运行测试
↓
模拟用户验收
↓
真实用户试用
↓
版本迭代
```

## 3. 模拟用户验收包含什么

### 1. 用户角色模拟

为产品设定 2-3 个用户角色。

示例：

| 用户 | 背景 | 目标 |
|---|---|---|
| 新用户 | 第一次打开产品 | 理解产品是做什么的 |
| 核心用户 | 有明确任务 | 完成核心流程 |
| 回访用户 | 已经使用过 | 找到上次保存或操作结果 |

### 2. 任务路径模拟

为每个用户设计任务。

示例：

```text
任务 1：
你是第一次打开产品的用户，请找到一个内容项，进入详情，执行核心动作，并找到动作结果。

期望路径：
首页 → 详情页 → 执行核心动作 → 结果页 / 状态页查看
```

### 3. 文档一致性检查

验收时不只看体验，还要检查实现是否符合文档：

- 用户看到的页面是否来自页面清单
- 用户动作是否来自用户流程
- 页面字段是否来自数据规范
- 功能是否来自 PRD
- 交互结果是否满足验收标准

### 4. 体验检查

检查：

- 用户是否知道下一步点哪里
- 页面标题是否清楚
- 按钮文案是否清楚
- 空状态是否有引导
- 错误状态是否有重试
- 用户完成任务是否需要猜

### 5. 逻辑检查

检查：

- 用户动作后是否有结果
- 取消动作后状态是否恢复
- 返回后状态是否一致
- 本地存储或数据状态是否按预期保留
- 页面跳转是否符合预期

## 4. 模拟用户提示词

```text
请你模拟一个真实用户，使用这个产品，并检查实现是否符合文档。

项目总说明：
【粘贴项目总说明】

PRD：
【粘贴产品需求文档】

页面清单：
【粘贴页面清单】

用户流程：
【粘贴用户流程】

数据规范：
【粘贴数据规范】

当前实现说明：
【粘贴 Agent 输出】

请从以下角度检查：
1. 第一次使用是否看得懂？
2. 用户是否知道下一步操作？
3. 页面文案是否容易理解？
4. 核心流程是否闭环？
5. 用户动作后是否有结果？
6. 当前实现是否符合页面清单？
7. 当前实现是否符合用户流程？
8. 当前实现是否使用了数据规范中的字段？
9. 是否有逻辑断点？
10. 如果你是目标用户，你会在哪里放弃？

请输出：
- 用户视角结论
- 文档一致性结论
- 主要卡点
- 必须修改项
- 可选优化项
- 是否适合交给真实用户试用
```

## 5. 人工模拟步骤

非技术学员也可以自己做：

1. 打开产品。
2. 不看代码，只按普通用户方式操作。
3. 完成一个任务。
4. 记录卡住的地方。
5. 对照文档检查是否符合 PRD、页面清单、用户流程和数据规范。
6. 把记录交给审查模型分析。
7. 转成下一轮任务。

## 6. 通过标准

满足下面条件，才可以交给真实用户试用：

- 第一次打开能理解这是做什么的
- 用户能在 1 分钟内完成核心流程
- 没有明显死路
- 用户动作后有结果承接
- 空状态和错误状态有解释
- 关键按钮文案清楚
- 实现能回溯到文档
- 模拟用户没有发现严重逻辑断点

## 7. 一句话原则

```text
验收不是问 Agent 做没做，
而是问用户能不能用、实现能不能回溯到文档。
```
