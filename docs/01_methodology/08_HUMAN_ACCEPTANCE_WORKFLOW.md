# 模拟用户验收机制

## 1. 为什么需要模拟用户验收

产品最终交付给人，不是交付给 AI。

代码能跑，不代表用户能用；功能完成，不代表体验正确。

所以在最终交付前，需要增加一个步骤：

```text
模拟用户验收
```

它用于模拟真实用户的使用路径、理解方式和操作逻辑，检查交付内容是否准确。

## 2. 验收位置

完整流程应该是：

```text
产品验证
↓
PRD
↓
页面清单
↓
任务拆分
↓
AI 编码
↓
代码审查
↓
运行测试
↓
模拟用户验收
↓
真实用户试用
↓
版本迭代
```

## 3. 模拟用户验收包含什么

### 1. 用户角色模拟

为产品设定 2-3 个用户角色。

示例：

| 用户 | 背景 | 目标 |
|---|---|---|
| 小白用户 | 不懂技术，只想找地点 | 找到一个周末可去的地点并收藏 |
| 内容用户 | 喜欢看攻略 | 查看地点详情和注意事项 |
| 回访用户 | 已经收藏过地点 | 快速找到自己的收藏 |

### 2. 任务路径模拟

为每个用户设计任务。

示例：

```text
任务 1：
你是第一次打开小程序的用户，请找到一个适合周末去的地点，并收藏它。

期望路径：
首页 → 详情页 → 点击收藏 → 收藏页查看
```

### 3. 体验检查

检查：

- 用户是否知道下一步点哪里
- 页面标题是否清楚
- 按钮文案是否清楚
- 空状态是否有引导
- 错误状态是否有重试
- 用户完成任务是否需要猜

### 4. 逻辑检查

检查：

- 收藏后收藏页是否出现
- 取消收藏后是否消失
- 返回首页后状态是否一致
- 本地存储后重新打开是否保留
- 页面跳转是否符合预期

## 4. AI 模拟用户提示词

```text
请你模拟一个真实用户，使用这个产品。

产品说明：
【粘贴项目总说明】

页面清单：
【粘贴页面清单】

核心流程：
【粘贴用户流程】

请从以下角度检查：
1. 第一次使用是否看得懂？
2. 用户是否知道下一步操作？
3. 页面文案是否容易理解？
4. 核心流程是否闭环？
5. 是否有逻辑断点？
6. 是否有让用户困惑的地方？
7. 如果你是目标用户，你会在哪里放弃？

请输出：
- 用户视角结论
- 主要卡点
- 必须修改项
- 可选优化项
- 是否适合交给真实用户试用
```

## 5. 人工模拟步骤

非技术学员也可以自己做：

1. 打开产品。
2. 不看代码，只按普通用户方式操作。
3. 完成一个任务。
4. 记录卡住的地方。
5. 把记录交给 AI 分析。
6. 转成下一轮 TASK。

## 6. 通过标准

满足下面条件，才可以交给真实用户试用：

- 第一次打开能理解这是做什么的
- 用户能在 1 分钟内完成核心流程
- 没有明显死路
- 空状态和错误状态有解释
- 关键按钮文案清楚
- AI 模拟用户没有发现严重逻辑断点
