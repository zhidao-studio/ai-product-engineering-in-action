# 核心原则

## 1. 原则总览

文档驱动的 AI 产品交付方法论有 5 条核心原则：

```text
一层只回答一个问题
造 harness 不能跳层
用漏测法检查漏洞
非技术人必须能验证
出问题先修 harness
```

## 2. 原则 1：一层只回答一个问题

每一层 harness 都必须回答一个清晰问题。

| 层 | 回答的问题 | 阻止的 AI 行为 |
|---|---|---|
| 产品定义 | 做什么 / 不做什么 | 阻止 AI 自己扩功能 |
| 设计规格 | 长什么样 | 阻止 AI 自己定风格 |
| 工程规格 | 怎么实现 | 阻止 AI 自己选方案 |
| 任务执行 | 这次只做什么 | 阻止 AI 一次做太多 |
| 验证 | 怎么算做对了 | 阻止 AI 自己说完成了 |
| 体验验收 | 用户能不能用 | 阻止只看代码不看体验 |
| 交付迭代 | 怎么上线和怎么改 | 阻止上线后随意变更 |

如果一份文档说不清它在阻止 AI 做什么，就不该保留。

## 3. 原则 2：造 harness 不能跳层

推荐顺序：

```text
产品定义
↓
设计规格
↓
工程规格
↓
任务执行
↓
验证
↓
体验验收
↓
交付迭代
```

每一层都是下一层的输入约束。

跳层会产生约束真空，AI 就会自由发挥。

## 4. 原则 3：用漏测法检查漏洞

交给 AI 之前，反向检查：

```text
AI 可能决定的每一件事，是否都有文档约束？
```

示例：

| AI 可能决定的内容 | 应该由谁约束 |
|---|---|
| 颜色 | 设计规格 |
| 卡片有没有图 | 设计原型 |
| 用什么技术栈 | 工程规格 |
| 字段叫什么 | 数据规范 |
| 接口返回什么 | API 规格 |
| 这次能改哪些文件 | 任务执行 Harness |
| 怎么判断通过 | 验证 Harness |

答不上来，就是 harness 漏洞。

## 5. 原则 4：非技术人必须能验证

Harness 不是给 AI 自己看的。

非技术人应该能根据文档判断：

- 这个需求该不该做
- AI 有没有扩功能
- 页面是否符合设计
- 功能是否满足验收标准
- 问题应该回到哪份文档修

如果必须看懂代码才能判断，harness 就不合格。

## 6. 原则 5：出问题先修 harness

当 AI 产出不符合预期时，先判断：

```text
是文档没有写清楚？
还是 AI 没遵守文档？
```

如果是文档没有写清楚，优先补 harness。

如果是 AI 没遵守文档，就缩小任务范围或调整提示词。

不要绕过 harness 直接手工改代码。

## 7. 一句话总结

```text
Harness 的价值不是文档多，
而是每一层都能阻止一种 AI 失控行为。
```
