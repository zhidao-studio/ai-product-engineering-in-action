# AI / Agent 工具分工表

## 1. 核心原则

本课程不绑定某一个 AI 编码工具或 Agent。

课程关注的是：

```text
文档控制
任务边界
Agent 执行
审查验收
持续维护
```

而不是某一个具体工具名称。

## 2. 工具角色总表

| 角色 | 可选工具 | 主要工作 | 产物 |
|---|---|---|---|
| 产品 / 文档助手 | ChatGPT / Kimi / DeepSeek / 豆包等通用大模型 | 整理想法、生成和修正文档、拆需求 | 项目总说明、PRD、页面清单、用户流程、数据规范 |
| 代码生成 Agent | Codex / Cursor / Windsurf / Kimi Coding / Trae / MarsCode / GitHub Copilot | 按任务读取文档、修改文件、修复问题 | 代码变更、修改说明、验证方式、风险点 |
| 审查 Agent | DeepSeek / ChatGPT / 其他审查模型 | 检查实现是否符合文档、任务和验收标准 | 审查报告、风险清单、必须修改项 |
| 长文档资料助手 | Kimi / ChatGPT / 其他长上下文模型 | 整理资料、竞品、课程文案、长文档 | 资料摘要、竞品分析、文案 |
| 学员 / 产品负责人 | 学员本人 | 决定做什么、不做什么、是否通过验收 | 最终判断、任务优先级、发布决定 |

## 3. 最重要的一句话

```text
文档工程定义边界，
Agent 在边界内执行，
审查模型检查偏差，
学员做最终判断。
```

## 4. 为什么不强绑定某个工具

Codex、Cursor、Windsurf、Kimi Coding、Trae、Copilot 等都可以作为代码生成 Agent，但不应该是课程唯一工具。

原因：

1. 学员所在地区和账号条件不同。
2. 不同学员可用工具不同。
3. 工具会变化，但文档控制流程应该稳定。
4. 课程目标是教非技术学员控制需求实现，而不是只教某一个工具按钮。

## 5. 选择代码生成 Agent 的最低标准

只要工具满足下面条件，就可以接入本课程流程：

- 能读取项目文档。
- 能读取项目目录。
- 能根据任务修改文件。
- 能说明修改了哪些文件。
- 能按报错修复问题。
- 能接受允许修改和禁止修改范围。
- 能接受“只做一个任务”的约束。
- 能按验收标准输出验证方式。

## 6. 不建议的用法

不要这样做：

```text
同时让 3 个 Agent 改同一个任务。
```

原因：

- 容易改乱文件。
- 难以定位问题。
- 学员无法判断谁改对了。
- 工程不可维护。

建议：

```text
一个任务只交给一个执行 Agent。
完成后再交给审查 Agent 检查。
```

## 7. 学员自己的分工表

学员可以这样填写：

| 工作 | 我使用的工具 | 我自己要检查什么 |
|---|---|---|
| 整理产品想法 |  | 是否符合真实需求 |
| 写项目总说明 |  | 是否边界清楚 |
| 写产品需求文档 |  | 是否范围过大 |
| 拆页面清单 |  | 是否页面过多 |
| 写用户流程 |  | 是否动作后有结果 |
| 写数据规范 |  | 字段是否足够且不多余 |
| 写任务清单 |  | 是否每个任务只做一件事 |
| Agent 实现 |  | 是否只完成当前任务 |
| 修复问题 |  | 是否没有无关修改 |
| 代码审查 |  | 是否能回溯到文档 |
| 模拟用户验收 |  | 用户是否能看懂和完成流程 |

## 8. 一句话原则

```text
不是绑定工具，
而是绑定文档控制流程。
```
