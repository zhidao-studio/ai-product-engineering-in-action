# AI 工具分工表

## 1. 核心原则

本课程不绑定某一个 AI 编码工具。

课程关注的是：

```text
分工
流程
任务
验收
维护
```

而不是某一个具体工具名称。

## 2. 工具角色总表

| 角色 | 可选工具 | 主要工作 | 产物 |
|---|---|---|---|
| 产品设计助手 | ChatGPT / Kimi / DeepSeek / 豆包等通用大模型 | 整理想法、拆需求、写 PRD、拆页面 | 项目总说明、产品需求文档、页面清单 |
| AI 编码工具 | Codex / Cursor / Windsurf / Kimi Coding / Trae / MarsCode / GitHub Copilot | 读取工程、修改代码、修复问题、执行任务 | 代码变更、修改说明、验证方式 |
| 代码审查助手 | DeepSeek / ChatGPT / 其他审查模型 | 检查代码、逻辑、边界、风险 | 代码审查报告、风险清单 |
| 长文档资料助手 | Kimi / ChatGPT / 其他长上下文模型 | 整理资料、竞品、课程文案、长文档 | 资料摘要、竞品分析、文案 |
| 学员 / 产品负责人 | 学员本人 | 决定做什么、不做什么、是否通过验收 | 最终判断、任务优先级、发布决定 |

## 3. 最重要的一句话

```text
大模型想清楚，
AI 编码工具写出来，
审查模型查问题，
学员做决定。
```

## 4. 为什么不强绑定 Codex

Codex 可以作为 AI 编码工具之一，但不应该是课程唯一工具。

原因：

1. 学员所在地区和账号条件不同。
2. 不同学员可能更熟悉 Cursor、Windsurf、Kimi Coding、Trae、Copilot 等工具。
3. 工具会变化，但工程流程应该稳定。
4. 课程目标是教非技术学员管理 AI 开发过程，而不是只教某一个工具按钮。

## 5. 选择 AI 编码工具的最低标准

只要工具满足下面条件，就可以接入本课程流程：

- 能读取项目目录。
- 能根据任务修改文件。
- 能说明修改了哪些文件。
- 能按报错修复问题。
- 能遵守 `AGENTS.md`。
- 能接受“只做一个任务”的约束。

## 6. 不建议的用法

不要这样做：

```text
同时让 3 个 AI 编码工具改同一个功能。
```

原因：

- 容易改乱文件。
- 难以定位问题。
- 学员无法判断谁改对了。
- 工程不可维护。

建议：

```text
第一版只选一个主力 AI 编码工具。
```
