# 如何构造产品交付 Harness

## 1. 这篇文档解决什么问题

上一篇讲的是：

```text
Harness 里有哪些层。
```

这篇讲的是：

```text
怎么造这个 Harness。
```

也就是文档工程本身的方法论。

构造 Harness 的目标不是写更多文档，而是建立一套能约束 AI 交付的系统。

## 2. 一句话总结

```text
逐层回答问题，
反向检查漏洞，
确保非技术人可验证，
出问题先修 Harness，不直接改代码。
```

## 3. 原则 1：每一层只回答一个问题

造 Harness 之前，先问：

```text
这份文档在阻止 AI 做什么？
```

| 层 | 回答的问题 | 阻止的 AI 行为 |
|---|---|---|
| 产品定义 | 做什么 / 不做什么 | 阻止 AI 自己扩功能 |
| 设计规格 | 长什么样 | 阻止 AI 自己定风格 |
| 工程规格 | 怎么实现 | 阻止 AI 自己选方案 |
| 任务执行 | 这次只做什么 | 阻止 AI 一次做太多 |
| 验证 | 怎么算做对了 | 阻止 AI 自己说完成了 |
| 体验验收 | 真实用户能不能用 | 阻止只看代码不看体验 |
| 交付迭代 | 怎么上线和怎么改 | 阻止上线后随意变更 |

如果一份文档说不清它在阻止什么，这份文档就不该存在，或者还没设计清楚。

## 4. 原则 2：造 Harness 的顺序不能跳

推荐顺序：

```text
产品定义
↓
设计规格
↓
工程规格
↓
任务执行
↓
验证
↓
体验验收
↓
交付迭代
```

每一层都是下一层的输入约束。

跳层的结果就是 AI 从约束真空里自由发挥。

## 5. 造的时候从上往下，验的时候从下往上

构造 Harness：

```text
产品定义 → 设计规格 → 工程规格 → 任务 → 验证
```

验证 AI 产物：

```text
代码 / 页面 / 接口
↓
对应哪个任务？
↓
对应哪个工程规格？
↓
对应哪个设计规格？
↓
对应哪个 PRD 功能？
↓
是否仍符合产品定义？
```

如果产物无法回溯到某份文档，说明 Harness 有漏洞。

此时优先补文档，而不是直接改代码。

## 6. 原则 3：漏测法检查 Harness 有没有洞

造完 Harness、交给 AI 之前，做一次反向验证。

对 AI 每一个可能的产出维度，问：

```text
哪个文档约束了这个？
```

示例：

| AI 可能决定的内容 | 哪个文档管 | 如果没有文档管 |
|---|---|---|
| 用什么颜色 | 视觉风格 / 配色规范 | 设计规格漏洞 |
| 卡片有没有图 | 设计原型 / 页面规格 | 设计规格漏洞 |
| 用 Vue 还是原生 H5 | 技术架构 | 工程规格漏洞 |
| 报错时显示什么 | 用户流程 / 页面状态 / 接口规范 | 流程或接口漏洞 |
| 空状态长什么样 | 设计原型 / 页面状态 | 设计规格漏洞 |
| 按钮放在哪里 | 原型 / 布局规范 | 设计规格漏洞 |
| 字段叫什么 | 数据规范 | 工程规格漏洞 |
| 接口返回什么 | API 规格 | 接口漏洞 |
| 是否新增登录 | PRD 不做清单 | 产品定义漏洞 |

如果能回答出：

```text
XX 文档的 YY 条管这个。
```

说明这个维度被 Harness 控住了。

如果答不上来，就是漏洞，先补文档。

## 7. 原则 4：Harness 的质量标准不是全，而是非技术人能验证

Harness 最终是给人用的，不是只给 AI 看的。

判断 Harness 好不好，问三个问题：

```text
不看代码，只看文档，能不能判断 AI 做对了没有？

发现一个 bug，能不能追溯到是哪份文档没写清楚？

不理解代码，也能不能决定这个需求该不该做？
```

如果三个答案都是“能”，这个 Harness 基本合格。

## 8. 原则 5：Harness 本身也要可迭代

第一版 Harness 不需要完美，但需要可定位问题。

当 AI 产出不符合预期时，不要马上手动改代码。

先判断：

```text
AI 做错了
↓
是文档没写清楚，还是 AI 没遵守文档？
├─ 文档没写清楚 → 补文档，也就是修 Harness
└─ AI 没遵守文档 → 缩小任务范围，或者换提示方式
```

手动改代码等于绕过 Harness。

如果绕过 Harness，下次 AI 还会犯同样的错。

## 9. 构造 Harness 的操作步骤

### Step 1：写产品定义 Harness

输出：

- 产品验证报告
- 项目总说明
- PRD

检查：

- 做什么清楚吗？
- 不做什么清楚吗？
- 验收标准能是 / 否判断吗？

### Step 2：写设计规格 Harness

输出：

- 视觉风格定义
- 配色规范
- 组件样式规范
- 布局规范
- 原型 / 参考图

检查：

- 页面长什么样是否明确？
- 空状态 / 错误状态是否明确？
- 按钮、卡片、表单是否明确？
- 高保真参考是否足够约束前端？

### Step 3：写工程规格 Harness

输出：

- 数据规范
- 前端页面规格
- 后端 API 规格
- 数据库规格
- 技术架构

检查：

- 前后端字段是否一致？
- 接口是否来自 PRD？
- 技术选型是否被约束？
- 目录和文件职责是否明确？

### Step 4：写任务执行 Harness

输出：

- 任务清单
- 允许修改范围
- 禁止修改范围
- 输出格式要求

检查：

- 一次只放开一个最小任务吗？
- 任务是否可独立验收？
- 修改范围是否具体到文件？

### Step 5：写验证 Harness

输出：

- 代码审查规则
- 功能测试用例
- 视觉比对规则
- 数据一致性检查
- 文档追溯检查

检查：

- 怎么判断做对了？
- 谁来确认通过？
- 是否需要截图比对？
- 是否能发现越界实现？

### Step 6：写体验验收 Harness

输出：

- 用户验收测试脚本
- 体验评价标准
- 边缘场景用例
- 多设备测试清单

检查：

- 用户是否能不经解释完成核心流程？
- 空状态和错误状态是否能看懂？
- 移动端和不同浏览器是否可用？

### Step 7：写交付与迭代 Harness

输出：

- 部署检查清单
- 变更协议
- 反馈分类规则
- 迭代记录

检查：

- 需求变更是否先改文档？
- Bug 是否能定位到文档或任务？
- 反馈是否先分类再进入实现？

## 10. 最小可用 Harness

第一版不必一次写完所有高级文档。

但至少要有：

```text
项目总说明
PRD
设计原型说明
数据规范
技术架构
任务清单
验证清单
体验验收脚本
变更协议
```

缺这些，AI 很容易失控。

## 11. 最终原则

```text
Harness 不是越多越好，
而是每一层都能阻止一种 AI 失控行为。
```

```text
如果一个 AI 产出维度没有文档约束，
它就是 Harness 漏洞。
```
