# AI 驱动交付 Harness 架构

## 1. 这篇文档解决什么问题

本课程要解决的不是“某个项目需要哪些文档”，而是：

```text
如何建立一套可复用的产品交付 Harness，
让 AI 在约束内起草、执行、验证，
让人负责确认决策和最终判断。
```

Harness 的核心作用是：

```text
约束 AI 能做什么，
阻止 AI 自由发挥，
让每一次交付都能被验证和追溯。
```

## 2. Harness 和普通文档的区别

普通文档常常只是说明：

```text
这个产品是什么。
```

Harness 文档必须能约束：

```text
AI 不能做什么。
AI 这次只能做什么。
AI 做完后怎么判断对错。
```

所以 Harness 不是说明书，而是约束系统。

## 3. 七层 Harness 总览

| 层级 | 名称 | 核心问题 | 主要制品 |
|---|---|---|---|
| 1 | 产品定义 Harness | 做什么 / 不做什么 | 产品验证报告、项目总说明、PRD |
| 2 | 设计规格 Harness | 长什么样 | 视觉风格、配色、组件、布局、原型 / 参考图 |
| 3 | 工程规格 Harness | 怎么实现 | 数据规范、前端规格、后端 API、数据库规格、技术架构 |
| 4 | 任务执行 Harness | 这次只做什么 | 任务清单、允许修改范围、禁止修改范围、输出格式 |
| 5 | 验证 Harness | 怎么判断做对了 | 代码审查规则、功能测试、视觉比对、数据一致性、文档追溯 |
| 6 | 体验验收 Harness | 真实用户能不能用 | 用户验收脚本、体验评价标准、边缘场景、多设备清单 |
| 7 | 交付与迭代 Harness | 怎么交付和怎么改 | 部署清单、变更协议、反馈分类、迭代记录 |

## 4. 第 1 层：产品定义 Harness

### 目的

锁死“做什么”和“不做什么”。

AI 和人都不能在这一层自由发挥。

### 制品

| 制品 | 约束什么 | AI 角色 | 人工角色 |
|---|---|---|---|
| 产品验证报告 | 证明是否值得做 | 辅助回答验证问题 | 判断是否进入 |
| 项目总说明 | 产品边界、目标用户、成功标准 | 起草 | 确认 |
| PRD | 功能清单、验收标准、不做清单 | 起草 | 确认 |

### Harness 机制

PRD 里的“不做清单”是 Harness 的否定约束。

验收标准是 Harness 的断言。

每个功能必须能用“是 / 否”判断是否完成。

## 5. 第 2 层：设计规格 Harness

### 目的

锁死“长什么样”，填补产品定义和代码之间的视觉真空。

### 制品

| 制品 | 约束什么 | AI 角色 | 人工角色 |
|---|---|---|---|
| 视觉风格定义 | 整体调性、参考设计 | 生成风格方案 | 选择 / 调整 |
| 配色规范 | 主色、辅色、语义色 | 生成配色方案 | 确认 |
| 组件样式规范 | 按钮、卡片、表单、导航 | 生成组件规格 | 确认 |
| 布局规范 | 间距、栅格、响应式断点 | 生成布局骨架 | 确认 |
| 原型 / 参考图 | 可视化的页面形态 | 生成高保真原型 | 确认或提供参考 |

### Harness 机制

设计规格可以有两种形态：

```text
文档形态：设计规范文档
原型形态：截图 / 高保真 mockup / 参考图
```

原型形态通常比纯文字约束更强。

如果用户能提供真实参考截图，约束力最强。

## 6. 第 3 层：工程规格 Harness

### 目的

锁死“怎么实现”，把产品和设计翻译成前后端各自的工程约束。

### 制品

| 制品 | 约束什么 | 面向 |
|---|---|---|
| 数据规范 | 字段定义、类型、关系、存储方式 | 前后端共享 |
| 前端页面规格 | 页面结构、组件树、状态管理 | 前端 |
| 后端 API 规格 | path / method / request / response | 后端 |
| 数据库规格 | 表结构、索引、关系 | 后端 |
| 技术架构 | 文件职责、目录结构、依赖边界 | 前后端各自 |

### Harness 机制

前后端通过数据规范和 API 规格对接。

它们相当于工程中的 interface contract。

AI 编码任务必须引用这些规格，不能自行发明字段、接口或技术方案。

## 7. 第 4 层：任务执行 Harness

### 目的

锁死“AI 这次只能做什么”。

一次只放开一个最小执行窗口。

### 制品

| 制品 | 约束什么 |
|---|---|
| 任务清单 | 编号、目标、依赖文档、任务类型 |
| 允许修改范围 | 具体到文件名或目录 |
| 禁止修改范围 | 文件和能力边界 |
| 验收标准 | 可检查的 pass / fail 条件 |
| 输出格式要求 | 修改文件列表、实现说明、风险点 |

### Harness 机制

每个任务就是一个独立、可运行、可判断的测试单元。

AI 每次执行后必须自检：

```text
是否只完成了当前任务？
是否只改了允许的文件？
是否没有扩大范围？
```

## 8. 第 5 层：验证 Harness

### 目的

锁死“怎么判断做对了”。

不能让 AI 自己说自己完成了。

### 制品

| 制品 | 验证什么 | AI 角色 | 人工角色 |
|---|---|---|---|
| 代码审查规则 | 代码是否对应文档、是否可维护 | 审查 Agent | 确认审查结果 |
| 功能测试用例 | 是否满足 PRD 验收标准 | 执行或辅助生成 | 抽查 |
| 视觉比对 | 实际页面 vs 设计规格 / 原型 | 截图比对 / VLM | 确认 |
| 数据一致性 | 字段、类型、存储是否符合规范 | 检查 | 确认 |
| 文档追溯 | 代码能否对应到文档 | 检查 | 确认 |

### Harness 机制

AI 验证不是“AI 说通过就通过”。

AI 只能按规则产出验证报告，人工做最终判断。

## 9. 第 6 层：体验验收 Harness

### 目的

锁死“真实用户能不能用”。

这区别于代码层面的“能不能跑”。

### 制品

| 制品 | 验证什么 | AI 角色 | 人工角色 |
|---|---|---|---|
| 用户验收测试脚本 | 真实用户操作路径 | 生成脚本 | 找真人执行 |
| 体验评价标准 | 是否需要解释、是否能完成核心流程 | 模拟走查 | 最终判断 |
| 边缘场景用例 | 空状态、错误状态、极端操作 | 生成用例 | 确认 |
| 多设备测试清单 | 屏幕尺寸、浏览器、系统 | 生成清单 | 执行或抽查 |

### Harness 机制

AI 可以做模拟用户走查。

但最终体验判断必须由真人做。

因为 AI 很难可靠判断按钮位置、信息层级和真实心理负担是否舒服。

## 10. 第 7 层：交付与迭代 Harness

### 目的

锁死“怎么交付”和“怎么改”。

产品上线后也必须可控。

### 制品

| 制品 | 约束什么 |
|---|---|
| 部署检查清单 | 上线前必须检查什么 |
| 变更协议 | 任何修改必须先改文档再改代码 |
| 反馈分类规则 | bug / 体验问题 / 范围变更 / 新需求 |
| 迭代记录 | 每次变更关联到哪个任务、哪份文档 |

### Harness 机制

变更协议是回归保护。

反馈分类是输入过滤。

只有被文档控制的需求才能进入实现。

## 11. 全景流程

```text
人工决策          AI 辅助           AI 执行           AI 验证          人工最终判断
──────────────────────────────────────────────────────────────────────────────→

[1] 产品定义 ──→ AI 起草 ──→ 人工确认
      ↓
[2] 设计规格 ──→ AI 生成方案 / 原型 ──→ 人工确认
      ↓
[3] 工程规格 ──→ AI 起草 API / 数据结构 ──→ 人工确认
      ↓
[4] 任务执行 ──→ AI 按任务编码 ──→
      ↓                              ↓
[5] 验证    ←── AI 审查代码 ←── 人工确认审查结果
      ↓
[6] 体验验收 ←── AI 模拟走查 ←── 人工找真人验收
      ↓
[7] 交付迭代 ──→ 反馈 → 文档 → 任务 → 编码 → 验证 → 交付
```

## 12. 人和 AI 的边界

人在每个阶段主要做两类事：

```text
确认决策
做最终判断
```

AI 可以做：

```text
起草
生成
执行
检查
模拟
整理
```

但 AI 不能替代：

```text
是否值得做
是否符合用户真实需求
是否接受这个设计
是否可以发布
是否继续投入
```

## 13. 一句话原则

```text
Harness 不是文档堆叠，
而是一套让 AI 可控交付的约束系统。
```
