# Harness 分层结构

## 1. 为什么要分层

AI 产品交付不是一句提示词，也不是一份 PRD。

它需要多层约束共同工作。

每一层都控制一个维度，防止 AI 在该维度自由发挥。

## 2. 七层结构

| 层级 | 名称 | 控制问题 | 主要产物 |
|---:|---|---|---|
| 1 | 产品定义 Harness | 做什么 / 不做什么 | 产品验证、项目总说明、PRD |
| 2 | 设计规格 Harness | 长什么样 | 风格、配色、组件、布局、原型 |
| 3 | 工程规格 Harness | 怎么实现 | 数据、前端、后端、接口、技术架构 |
| 4 | 任务执行 Harness | 这次只做什么 | 任务清单、允许范围、禁止范围 |
| 5 | 验证 Harness | 怎么算做对了 | 代码审查、功能测试、视觉比对、数据检查 |
| 6 | 体验验收 Harness | 用户能不能用 | 用户验收脚本、体验标准、边缘场景 |
| 7 | 交付迭代 Harness | 怎么上线和怎么改 | 部署清单、变更协议、反馈分类、迭代记录 |

## 3. 第 1 层：产品定义 Harness

目的：锁定产品边界。

回答：

```text
做什么？
给谁用？
不做什么？
怎样算成功？
```

主要产物：

- 产品验证报告
- 项目总说明
- PRD

## 4. 第 2 层：设计规格 Harness

目的：锁定视觉和交互表达。

回答：

```text
页面长什么样？
模块怎么排？
按钮在哪里？
空状态和错误状态怎么展示？
```

主要产物：

- 视觉风格定义
- 配色规范
- 组件样式规范
- 布局规范
- 原型或参考图

## 5. 第 3 层：工程规格 Harness

目的：锁定工程实现边界。

回答：

```text
数据字段是什么？
前端怎么组织？
后端是否需要？
接口如何定义？
技术方案有哪些限制？
```

主要产物：

- 数据规范
- 前端规格
- 后端规格
- API 规格
- 技术架构

## 6. 第 4 层：任务执行 Harness

目的：锁定 AI 本次执行窗口。

回答：

```text
这次只做什么？
能改哪些文件？
不能改哪些文件？
完成后输出什么？
```

主要产物：

- 任务清单
- 允许修改范围
- 禁止修改范围
- 输出格式要求

## 7. 第 5 层：验证 Harness

目的：锁定怎么判断做对了。

回答：

```text
代码是否符合文档？
功能是否满足验收标准？
视觉是否符合设计？
数据是否一致？
```

主要产物：

- 代码审查规则
- 功能测试用例
- 视觉比对清单
- 数据一致性检查
- 文档追溯检查

## 8. 第 6 层：体验验收 Harness

目的：锁定真实用户能不能用。

回答：

```text
用户是否看得懂？
用户是否知道点哪里？
用户是否能完成核心流程？
异常情况是否可理解？
```

主要产物：

- 用户验收测试脚本
- 体验评价标准
- 边缘场景用例
- 多设备测试清单

## 9. 第 7 层：交付迭代 Harness

目的：锁定上线和变更流程。

回答：

```text
上线前检查什么？
反馈如何分类？
变更是否先改文档？
每次迭代如何追溯？
```

主要产物：

- 部署检查清单
- 变更协议
- 反馈分类规则
- 迭代记录

## 10. 一句话总结

```text
七层 Harness 不是为了复杂，
而是为了让 AI 产品交付的每个关键维度都有约束、有模板、有检查。
```
