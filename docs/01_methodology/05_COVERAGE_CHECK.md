# Harness 覆盖检查

## 1. 定义

覆盖检查用于判断 harness 是否覆盖了 AI 交付中的主要决策点。

核心问题：

```text
AI 可能产出的每个维度，是否都有对应约束？
```

没有对应约束的维度，需要补充模板或 checklist。

## 2. 覆盖检查表

| AI 产出维度 | 对应约束 |
|---|---|
| 是否值得做 | 产品定义 Harness |
| 是否新增功能 | PRD / 不做清单 |
| 页面数量 | 页面规格 |
| 页面视觉 | 设计规格 |
| 颜色 | 配色规范 |
| 按钮位置 | 原型 / 布局规范 |
| 空状态 | 页面状态 / 设计规格 |
| 字段名称 | 数据规范 |
| 技术选择 | 技术架构 |
| 接口响应 | API 规格 |
| 修改范围 | 任务执行 Harness |
| 通过标准 | 验证 Harness |
| 用户体验 | 体验验收 Harness |
| 上线变更 | 交付迭代 Harness |

## 3. 检查步骤

第一步：列出 AI 可能产出的内容。

第二步：为每一项找到约束文档。

第三步：找不到约束时，先补模板或 checklist，再进入执行。

## 4. 合格标准

- 主要产出维度都有约束。
- 每个任务都有允许范围和禁止范围。
- 每个功能都有可检查标准。
- 非技术人能根据文档判断是否通过。
- 问题能追溯到具体文档或 checklist。

## 5. 一句话原则

```text
AI 可能产出的内容，必须能被 harness 覆盖。
```
