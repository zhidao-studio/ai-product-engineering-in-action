# 版本迭代指南

## 1. 第一版完成后，不要马上加功能

先做三件事：

1. 找 3-5 个真实用户试用。
2. 观察他们能不能完成核心流程。
3. 记录他们卡住的地方。

## 2. 反馈分类

把反馈分成四类：

| 类型 | 说明 | 处理方式 |
|---|---|---|
| 阻塞问题 | 用户无法完成核心流程 | 必须优先修 |
| 体验问题 | 用户能完成，但不顺 | 第二优先级 |
| 新功能建议 | 用户想要更多功能 | 先记录，不马上做 |
| 个人偏好 | 只是个别人喜好 | 暂不处理 |

## 3. 是否加功能的判断

一个新功能要进入下一版，至少满足：

- 不破坏当前核心流程
- 至少 2 个目标用户提到
- 能用一个小任务实现
- 有明确验收标准

## 4. 是否砍功能的判断

满足以下任一条件，可以考虑砍掉：

- 没人使用
- 解释成本太高
- 维护成本太高
- 影响核心流程
- AI 多次实现失败

## 5. 版本节奏

建议：

```text
v0.1：能跑
v0.2：能用
v0.3：体验顺
v0.4：接真实数据
v1.0：对外发布
```

## 6. 每次迭代必须更新

- CHANGELOG
- TASKS
- REVIEW
- README 中的当前版本说明
