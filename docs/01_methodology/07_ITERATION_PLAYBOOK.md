# 版本迭代指南

## 1. 第一版完成后，不要马上加功能

先做三件事：

1. 找 3-5 个真实用户试用。
2. 观察他们能不能完成核心流程。
3. 记录他们卡住的地方。

注意：

```text
用户反馈不能直接进入代码，
必须先进入文档控制体系。
```

## 2. 反馈分类

把反馈分成四类：

| 类型 | 说明 | 处理方式 |
|---|---|---|
| 阻塞问题 | 用户无法完成核心流程 | 必须优先修 |
| 体验问题 | 用户能完成，但不顺 | 第二优先级 |
| 新功能建议 | 用户想要更多功能 | 先记录，不马上做 |
| 个人偏好 | 只是个别人喜好 | 暂不处理 |

## 3. 反馈进入文档的规则

不同反馈进入不同文档：

| 反馈内容 | 先改哪份文档 |
|---|---|
| 产品定位变了 | 项目总说明 |
| 功能范围变了 | PRD |
| 页面入口不清楚 | 页面清单 |
| 用户流程断了 | 用户流程 |
| 字段不够用 | 数据规范 |
| 文件职责不清 | 技术结构 |
| 需要实现修改 | 任务清单 |
| 实现不一致 | 文档到代码映射 / 验收清单 |

## 4. 是否加功能的判断

一个新功能要进入下一版，至少满足：

- 不破坏当前核心流程
- 至少 2 个目标用户提到
- 能被写进 PRD
- 能拆成一个或多个小任务
- 有明确验收标准
- 不突破第一版能力边界

## 5. 是否砍功能的判断

满足以下任一条件，可以考虑砍掉：

- 没人使用
- 解释成本太高
- 维护成本太高
- 影响核心流程
- Agent 多次实现失败
- 无法写出清楚验收标准

## 6. 版本节奏

建议：

```text
v0.1：文档骨架清楚
v0.2：核心文档完整
v0.3：图文课程可跟做
v0.4：文档控制体系完整
v0.5：样例工程可运行
v1.0：可公开系统学习
```

## 7. 每次迭代必须更新

- README 当前版本说明
- 版本路线图
- PRD
- 页面清单
- 用户流程
- 数据规范
- 技术结构
- 任务清单
- 文档到代码映射
- 验收清单
- CHANGELOG

## 8. 一句话原则

```text
所有变更先进入文档，
再进入任务，
最后才进入代码。
```
