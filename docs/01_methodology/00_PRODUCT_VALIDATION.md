# 产品想法验证

## 1. 为什么要先验证

Agent 可以让你更快实现需求，但也会让你更快实现一个没人需要的需求。

所以在写 PRD、拆任务、让 Agent 实现之前，要先做产品想法验证。

验证不是为了马上进入代码，而是为了确认：

```text
这个需求是否值得进入文档控制体系。
```

## 2. 验证目标

不是证明你的想法一定正确，而是尽早发现：

- 用户是否真实存在
- 痛点是否真实存在
- 用户现在怎么解决
- 竞品或替代方案是否已经解决
- 第一版是否值得做
- 第一版应该砍掉哪些功能
- 需求是否能被文档控制住

## 3. 产品验证六问

### 1. 用户是谁？

不要写“所有人”。

错误：

```text
适合所有喜欢生活的人。
```

正确：

```text
适合周末去城市周边钓鱼、露营、短途户外的人。
```

### 2. 痛点是什么？

用户现在遇到什么麻烦？

示例：

```text
想找能露营又能钓鱼的地方，但信息分散在社交平台、群聊和地图评论里，不知道哪些真实可去。
```

### 3. 用户现在怎么解决？

他们现在是否已经有替代方案？

示例：

```text
现在靠社交平台搜索、朋友推荐、群聊分享、地图评论、自己踩点。
```

### 4. 竞品是谁？

竞品不一定是 App，也可能是：

- 社交平台笔记
- 群聊
- 短视频
- 地图收藏夹
- 表格
- 朋友推荐

### 5. 你的第一版凭什么有价值？

第一版不要追求功能多，要有一个小价值。

示例：

```text
先把“可露营 + 可钓鱼 + 周末可去”的地点整理成可收藏列表。
```

### 6. 怎么验证有没有人要？

可以用很低成本验证：

- 发 3 条内容测试反馈
- 做一个表格收集地点
- 做一个群投票
- 给 5 个朋友试用
- 让用户收藏 3 个地点
- 看是否有人愿意持续反馈新地点

## 4. 验证结论模板

```text
目标用户：
真实痛点：
当前替代方案：
竞品/替代品：
第一版价值：
验证方式：
是否值得进入文档控制体系：
主要风险：
```

## 5. 进入文档控制体系的最低标准

满足下面 3 条，再进入项目总说明和 PRD：

- 能说清具体目标用户
- 能说清具体使用场景
- 能说清第一版只验证一个核心闭环

不满足就先不要写 PRD，更不要让 Agent 实现。

## 6. 从验证到文档

验证通过后，不是直接实现，而是进入：

```text
项目总说明
↓
产品需求文档
↓
页面清单
↓
用户流程
↓
数据规范
↓
技术结构
↓
任务清单
```

## 7. 一句话原则

```text
未经验证的想法，不进入文档。
未经文档控制的需求，不进入实现。
```
