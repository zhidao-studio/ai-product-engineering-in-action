# 产品想法验证

## 1. 为什么要先验证

AI 可以让你更快做出产品，但也会让你更快做出一个没人用的产品。

所以在写 PRD 和代码之前，要先做产品想法验证。

## 2. 验证目标

不是证明你的想法一定正确，而是尽早发现：

- 用户是否真实存在
- 痛点是否真实存在
- 用户现在怎么解决
- 竞品是否已经解决
- 第一版是否值得做
- 第一版应该砍掉哪些功能

## 3. 产品验证六问

### 1. 用户是谁？

不要写“所有人”。

错误：

```text
适合所有喜欢生活的人。
```

正确：

```text
适合上海周边周末去钓鱼、露营、短途户外的人。
```

### 2. 痛点是什么？

用户现在遇到什么麻烦？

示例：

```text
想找能露营又能钓鱼的地方，但信息分散在小红书、抖音、群聊和地图评论里，不知道哪些真实可去。
```

### 3. 用户现在怎么解决？

他们现在是否已经有替代方案？

示例：

```text
现在靠小红书搜索、朋友推荐、微信群分享、地图评论、自己踩点。
```

### 4. 竞品是谁？

竞品不一定是 App，也可能是：

- 小红书笔记
- 微信群
- 抖音视频
- 地图收藏夹
- Excel 表
- 朋友推荐

### 5. 你的第一版凭什么有价值？

第一版不要追求功能多，要有一个小价值。

示例：

```text
先把“可露营 + 可钓鱼 + 周末可去”的地点整理成可收藏列表。
```

### 6. 怎么验证有没有人要？

可以用很低成本验证：

- 发 3 条小红书内容
- 做一个 Notion / 飞书表格
- 做一个微信群投票
- 给 5 个朋友试用
- 让用户收藏 3 个地点
- 看是否有人愿意持续反馈新地点

## 4. 验证结论模板

```text
目标用户：
真实痛点：
当前替代方案：
竞品/替代品：
第一版价值：
验证方式：
是否值得进入开发：
主要风险：
```

## 5. 进入开发的最低标准

满足下面 3 条再进入开发：

- 能说清具体目标用户
- 能说清具体使用场景
- 能说清第一版只验证一个核心闭环

不满足就先不要写代码。
