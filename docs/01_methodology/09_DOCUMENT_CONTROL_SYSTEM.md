# 文档控制需求实现体系

## 1. 核心诉求

本课程的核心不是让 AI 写代码，而是：

```text
通过一整套完整文档，控制需求如何被实现。
```

代码只是最终产物。

真正要控制的是：

- 需求边界
- 功能范围
- 页面范围
- 交互流程
- 信息结构
- 数据字段
- 设计和原型
- 前端实现边界
- 后端实现边界
- 接口和联调边界
- Agent 执行范围
- 验收标准
- 变更记录

## 2. 一个重要修正：不是线性文档，而是联动约束

旧理解容易变成：

```text
PRD → 页面清单 → 用户流程 → 数据规范 → 技术结构 → 任务清单
```

这个顺序有帮助，但不完整。

真实产品工程里，下面这些内容必须一起考虑：

```text
功能范围
页面范围
交互流程
信息结构
数据字段
设计原型
前端结构
后端能力
接口边界
```

因为它们互相影响：

- 功能范围会影响页面范围。
- 页面范围会影响交互流程。
- 交互流程会影响数据字段。
- 数据字段会影响前端展示和后端接口。
- 设计原型会暴露页面状态和交互问题。
- 技术架构会反过来约束第一版能不能做。

所以本课程采用的是：

```text
先有产品边界，
再做需求设计联动，
再拆前端 / 后端 / 接口执行，
最后验收是否回到文档。
```

## 3. 文档控制体系总览

| 层级 | 文档 | 控制对象 | 不允许 Agent 自行决定的内容 |
|---|---|---|---|
| 产品层 | 项目总说明 | 产品是什么、给谁用、解决什么问题 | 目标用户、产品定位、第一版边界 |
| 需求层 | 产品需求文档 | 第一版做什么、不做什么 | 功能范围、优先级、验收标准 |
| 需求设计联动层 | 页面清单 / 用户流程 / 数据规范 / 设计原型 | 页面、流程、字段、状态、交互、视觉结构 | 页面数量、字段名、交互结果、页面状态、布局意图 |
| 工程架构层 | 技术结构 / 前端结构 / 后端结构 / 接口规范 | 工程拆分、文件职责、接口边界、存储方式 | 是否需要后端、接口形态、文件怎么分、能力边界 |
| 执行层 | 任务清单 | Agent 每次改什么 | 一次改多个任务、扩大范围、跨端乱改 |
| 映射层 | 文档到代码映射 | 代码来自哪份文档 | 代码凭空出现、字段随意发明、接口自行新增 |
| 验收层 | 审查和模拟用户验收 | 是否真正可交付 | Agent 自称完成但无人验证 |

## 4. 需求设计联动层：这些必须一起考虑

下面几类文档不能孤立写：

| 文档 | 主要控制 | 必须联动 |
|---|---|---|
| 页面清单 | 有哪些页面、每页目标 | PRD、用户流程、设计原型 |
| 用户流程 | 用户动作和结果闭环 | 页面清单、数据规范、设计原型 |
| 数据规范 | 字段、类型、状态、存储 | 页面展示、交互流程、接口规范 |
| 设计原型 | 页面布局、信息层级、状态表现 | 页面清单、用户流程、数据字段 |
| 高保真设计 | 视觉风格、组件样式、可读性 | 前端实现、组件拆分、验收标准 |

判断标准：

```text
页面上出现的每个字段，都应该来自数据规范。
页面上的每个动作，都应该来自用户流程。
页面上的每个模块，都应该来自页面清单或设计原型。
设计里的每个状态，都应该能被前端实现和验收。
```

## 5. 代码工程执行层：必须拆前端和后端

代码工程执行不能只写“Agent 实现”。

至少要拆成：

```text
前端执行
后端执行
接口 / 数据联调
部署 / 运行环境
```

### 5.1 前端执行控制什么

前端主要控制：

- 页面结构
- 组件拆分
- 交互逻辑
- 状态展示
- 样式实现
- 路由或页面切换
- 表单和输入校验
- 调用接口或读取本地数据

前端不能自行决定：

- 新增页面
- 新增字段
- 新增业务规则
- 自行改变用户流程
- 自行新增后端接口

### 5.2 后端执行控制什么

后端主要控制：

- 数据模型
- API 接口
- 业务规则
- 数据存储
- 权限和登录
- 任务调度
- 服务部署

第一版如果没有后端，也必须明确写出：

```text
第一版不做后端。
使用 Mock 数据或本地存储。
后端能力进入后续版本。
```

后端不能自行决定：

- 新增 PRD 没有定义的业务能力
- 新增前端没有使用的接口
- 自行改变字段含义
- 自行加入复杂权限、支付、消息、推荐等能力

### 5.3 接口 / 数据联调控制什么

接口和数据联调主要控制：

- 前端需要哪些字段
- 后端返回哪些字段
- 字段类型是否一致
- 空状态和错误状态如何返回
- 本地 Mock 如何迁移到真实接口
- 接口失败时前端如何展示

这部分必须由《接口规范》或《数据接口规范》控制。

## 6. 设计和原型为什么必须加入

如果没有设计和原型，文档容易只停留在文字层面。

非技术学员尤其容易漏掉：

- 页面信息层级
- 按钮位置
- 空状态
- 错误状态
- 加载状态
- 表单校验
- 用户动作后的反馈
- 移动端适配
- 高保真视觉风格

所以在进入前端实现前，至少需要一份轻量原型文档。

推荐顺序：

```text
页面清单
↓
用户流程
↓
低保真原型
↓
数据字段校验
↓
高保真设计
↓
前端实现任务
```

第一版不一定要用 Figma，但必须有：

- 页面布局草图
- 模块顺序
- 关键按钮
- 页面状态
- 核心文案
- 高保真风格说明或截图参考

## 7. Agent 在体系中的位置

Agent 可以替换。

可以是：

- 产品文档 Agent
- 原型设计 Agent
- 前端代码 Agent
- 后端代码 Agent
- 接口联调 Agent
- 审查 Agent

但 Agent 不能替代文档体系。

正确关系：

```text
文档工程负责控制。
不同 Agent 负责不同执行面。
学员负责判断。
```

## 8. 每次实现前必须回答的问题

任何一段实现开始前，都必须能回答：

```text
这个需求来自哪里？
它属于哪个功能编号？
它对应哪个页面？
它符合哪个用户流程？
它是否有原型或设计依据？
它使用哪些数据字段？
它属于前端任务、后端任务还是联调任务？
它允许修改哪些文件？
它禁止修改哪些文件？
它完成后怎么验收？
```

如果回答不了，就不能进入实现。

## 9. 每次实现后必须回答的问题

Agent 完成后，必须检查：

```text
它是否只完成当前任务？
它是否只修改允许文件？
它是否新增了未经允许的功能？
它是否新增了未经允许的依赖？
它是否符合 PRD？
它是否符合页面清单？
它是否符合用户流程？
它是否符合数据规范？
它是否符合原型 / 设计要求？
它是否符合前端 / 后端 / 接口边界？
它是否满足验收标准？
用户是否能完成核心流程？
```

## 10. 需求变更怎么处理

需求变更不能直接改代码。

正确流程：

```text
提出新需求
↓
判断是否属于当前版本范围
↓
判断影响哪些文档：PRD / 页面 / 流程 / 数据 / 原型 / 前端 / 后端 / 接口
↓
先改受影响文档
↓
更新任务清单
↓
再让对应 Agent 实现
↓
最后审查和验收
```

## 11. 最小闭环

一套完整文档控制体系，至少要包含：

```text
项目总说明
产品需求文档
页面清单
用户流程
数据规范
设计原型说明
技术结构
前端实现说明
后端实现说明（如果第一版不做后端，也要说明不做）
接口规范（如果第一版不做接口，也要说明用 Mock / 本地存储替代）
任务清单
文档到代码映射
Agent 实现指南
验收清单
```

缺其中关键环节，都可能导致实现不可控。

## 12. 一句话原则

```text
不是用 AI 替代文档，
而是用文档控制 AI；
不是让一个 Agent 做所有事，
而是让不同执行层在同一套文档边界内协作。
```
