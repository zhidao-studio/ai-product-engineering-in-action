# 文档控制需求实现体系

## 1. 核心诉求

本课程的核心不是让 AI 写代码，而是：

```text
通过一整套完整文档，控制需求如何被实现。
```

代码只是最终产物。

真正要控制的是：

- 需求边界
- 页面范围
- 用户流程
- 数据结构
- 技术结构
- Agent 执行范围
- 验收标准
- 变更记录

## 2. 为什么必须用文档控制实现

如果没有文档控制，常见问题是：

```text
用户一句想法
↓
Agent 自己理解
↓
Agent 自己扩功能
↓
Agent 自己决定技术方案
↓
代码看起来完成了
↓
但产品范围、页面、数据、交互都不可控
```

这不是产品工程，这是随机生成。

正确方式是：

```text
用户想法
↓
文档工程定义边界
↓
任务清单拆小
↓
Agent 按任务执行
↓
文档映射审查
↓
模拟用户验收
```

## 3. 文档控制体系总览

| 层级 | 文档 | 控制对象 | 不允许 Agent 自行决定的内容 |
|---|---|---|---|
| 产品层 | 项目总说明 | 产品是什么、给谁用、解决什么问题 | 目标用户、产品定位、第一版边界 |
| 需求层 | 产品需求文档 | 第一版做什么、不做什么 | 功能范围、优先级、验收标准 |
| 页面层 | 页面清单 | 有哪些页面、每页做什么 | 页面数量、页面入口、页面状态 |
| 流程层 | 用户流程 | 用户如何完成核心任务 | 跳转关系、动作后的结果 |
| 数据层 | 数据规范 | 页面使用哪些字段 | 字段名、字段类型、存储方式 |
| 工程层 | 技术结构 | 文件职责和实现边界 | 是否引框架、文件怎么分、能力边界 |
| 执行层 | 任务清单 | Agent 每次改什么 | 一次改多个任务、扩大范围 |
| 映射层 | 文档到代码映射 | 代码来自哪份文档 | 代码凭空出现、字段随意发明 |
| 验收层 | 审查和模拟用户验收 | 是否真正可交付 | Agent 自称完成但无人验证 |

## 4. 文档不是说明书，而是控制器

很多人把文档当成说明书：

```text
代码写完后补一份文档。
```

本课程反过来：

```text
先写文档，
再用文档控制代码生成，
最后用文档验收代码。
```

所以文档不是“解释已经发生的事”，而是“限制接下来能发生什么”。

## 5. Agent 在体系中的位置

Agent 可以替换。

可以是：

- Codex
- Cursor
- Windsurf
- Kimi Coding
- Trae / MarsCode
- GitHub Copilot
- 其他能读取工程、修改文件、执行任务的模型或 Agent

但 Agent 不能替代文档体系。

正确关系：

```text
文档工程负责控制。
Agent 负责执行。
学员负责判断。
```

## 6. 每次实现前必须回答的问题

任何一段代码生成前，都必须能回答：

```text
这个需求来自哪里？
它属于哪个功能编号？
它对应哪个页面？
它符合哪个用户流程？
它使用哪些数据字段？
它允许修改哪些文件？
它禁止修改哪些文件？
它完成后怎么验收？
```

如果回答不了，就不能进入代码实现。

## 7. 每次实现后必须回答的问题

Agent 完成后，必须检查：

```text
它是否只完成当前任务？
它是否只修改允许文件？
它是否新增了未经允许的功能？
它是否新增了未经允许的依赖？
它是否符合页面清单？
它是否符合用户流程？
它是否符合数据规范？
它是否满足验收标准？
用户是否能完成核心流程？
```

## 8. 需求变更怎么处理

需求变更不能直接改代码。

正确流程：

```text
提出新需求
↓
判断是否属于第一版范围
↓
如果属于，先改 PRD / 页面清单 / 用户流程 / 数据规范
↓
更新任务清单
↓
再让 Agent 实现
↓
最后审查和验收
```

不允许：

```text
用户临时想到一个功能
↓
直接让 Agent 加到代码里
```

## 9. 最小闭环

一套完整文档控制体系，至少要包含：

```text
项目总说明
产品需求文档
页面清单
用户流程
数据规范
技术结构
任务清单
文档到代码映射
Agent 实现指南
验收清单
```

缺其中任何一环，都可能导致实现不可控。

## 10. 一句话原则

```text
不是用 AI 替代文档，
而是用文档控制 AI。
```
