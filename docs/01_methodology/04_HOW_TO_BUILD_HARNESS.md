# 如何构造 Harness

## 1. 这篇文档解决什么问题

前面说明了 harness 有哪些层。

这篇说明：

```text
怎么把这些层一步一步造出来。
```

目标不是写很多文档，而是构造一套能约束 AI 交付的系统。

## 2. 构造顺序

推荐顺序：

```text
产品定义
↓
设计规格
↓
工程规格
↓
任务执行
↓
验证
↓
体验验收
↓
交付迭代
```

不要跳层。

每一层都是下一层的输入约束。

## 3. Step 1：构造产品定义 Harness

目标：锁定做什么和不做什么。

需要产出：

- 产品验证报告
- 项目总说明
- PRD

必须回答：

```text
目标用户是谁？
解决什么问题？
第一版做什么？
第一版不做什么？
怎样算成功？
```

通过标准：

- 不做清单明确
- 功能有验收标准
- 非技术人能判断是否值得进入下一步

## 4. Step 2：构造设计规格 Harness

目标：锁定产品长什么样。

需要产出：

- 视觉风格定义
- 配色规范
- 组件样式规范
- 布局规范
- 原型或参考图

必须回答：

```text
页面整体风格是什么？
按钮、卡片、表单怎么长？
页面模块怎么排？
空状态、错误状态怎么展示？
```

通过标准：

- 前端不能自行决定视觉风格
- 页面状态有明确说明
- 有文档或图片约束高保真方向

## 5. Step 3：构造工程规格 Harness

目标：锁定怎么实现。

需要产出：

- 数据规范
- 前端规格
- 后端规格
- API 规格
- 技术架构

必须回答：

```text
字段有哪些？
前端页面和组件怎么拆？
后端是否需要？
接口怎么定义？
哪些技术方案不能用？
```

通过标准：

- 前后端字段一致
- API 来自真实功能需要
- 技术方案不会自行扩大范围

## 6. Step 4：构造任务执行 Harness

目标：锁定 AI 这次只能做什么。

需要产出：

- 任务清单
- 任务类型
- 允许修改范围
- 禁止修改范围
- 输出格式要求

必须回答：

```text
这次只做什么？
能改哪些文件？
不能改哪些文件？
完成后怎么输出？
```

通过标准：

- 一个任务只做一件事
- 修改范围具体
- 验收标准可检查

## 7. Step 5：构造验证 Harness

目标：锁定怎么判断做对了。

需要产出：

- 代码审查规则
- 功能测试用例
- 视觉比对规则
- 数据一致性检查
- 文档追溯检查

必须回答：

```text
功能是否对？
视觉是否对？
数据是否对？
代码是否能追溯到文档？
```

通过标准：

- AI 不能自己说完成
- 每个验证项有明确标准
- 人可以最终判断通过或不通过

## 8. Step 6：构造体验验收 Harness

目标：锁定用户能不能用。

需要产出：

- 用户验收测试脚本
- 体验评价标准
- 边缘场景用例
- 多设备测试清单

必须回答：

```text
用户能不能看懂？
用户能不能完成核心流程？
异常情况下用户知不知道怎么办？
```

通过标准：

- 不只看代码能不能跑
- 用户操作路径明确
- 体验问题能转成任务

## 9. Step 7：构造交付迭代 Harness

目标：锁定怎么上线和怎么改。

需要产出：

- 部署检查清单
- 变更协议
- 反馈分类规则
- 迭代记录

必须回答：

```text
上线前检查什么？
反馈怎么分类？
变更是否先改文档？
迭代怎么追溯？
```

通过标准：

- 反馈不能直接进入代码
- 变更必须关联文档和任务
- 每次交付都可追溯

## 10. 最小可用 Harness

第一版不必完美，但至少要有：

```text
项目总说明
PRD
设计规格
数据规范
工程规格
任务清单
验证清单
体验验收脚本
变更协议
```

## 11. 一句话总结

```text
构造 Harness 的过程，
就是逐层减少 AI 自由发挥空间的过程。
```
