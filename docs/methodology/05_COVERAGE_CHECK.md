# Harness 覆盖检查

## 1. 定义

覆盖检查用于判断 harness 是否覆盖了 AI 交付中的主要决策点。

核心问题：

```text
AI 可能产出的每个维度，是否都有对应约束？
```

没有对应约束的维度，需要补充模板或 checklist。

## 2. 覆盖检查表

| AI 产出维度 | 对应约束 | 常见缺口 |
|---|---|---|
| 是否值得做 | 产品定义 Harness | 没有产品验证 |
| 是否新增功能 | PRD / 不做清单 | AI 自行扩功能 |
| 页面数量 | 页面规格 / 设计规格 | 页面越做越多 |
| 页面视觉 | 设计规格 | 风格不一致 |
| 颜色 | 配色规范 | AI 自行配色 |
| 字体和字号 | 设计规格 / 组件规范 | 可读性不一致 |
| 组件样式 | 组件样式规范 | 按钮、卡片、表单各写各的 |
| 组件库选择 | 工程规格 / 技术边界 | AI 自行引入 UI 库 |
| 按钮位置 | 原型 / 布局规范 | 关键操作位置不稳定 |
| 布局和间距 | 布局规范 / 原型 | 页面密度混乱 |
| 响应式断点 | 布局规范 / 工程规格 | 手机和桌面表现不一致 |
| 动画和过渡行为 | 设计规格 | AI 自行加动效或无动效 |
| 空状态 | 页面状态 / 设计规格 | 空数据时无解释 |
| 错误状态 | 页面状态 / 体验验收 Harness | 错误时用户不知道怎么办 |
| 错误提示文案 | 体验验收 Harness / 设计规格 | 文案生硬或不可理解 |
| 表单校验规则 | 工程规格 / 体验验收 Harness | 输入错误无法判断 |
| 加载状态 | 设计规格 / 验证 Harness | 用户不知道是否正在处理 |
| 字段名称 | 数据规范 | 字段名不统一 |
| 字段类型 | 数据规范 | 前后端类型不一致 |
| 数据存储方式 | 工程规格 | AI 自行决定 localStorage / API / DB |
| 技术选择 | 工程规格 / 技术边界 | AI 自行换框架 |
| 接口路径 | API 规格 | 前后端路径不一致 |
| 接口请求参数 | API 规格 | 前端传参和后端不一致 |
| 接口响应字段 | API 规格 | 响应结构不一致 |
| 接口错误返回 | API 规格 / 体验验收 Harness | 前端无法展示错误 |
| 修改范围 | 任务执行 Harness | AI 一次改太多 |
| 输出格式 | 任务执行 Harness | AI 不说明改了什么 |
| 通过标准 | 验证 Harness | AI 自己说完成 |
| 用户核心流程 | 体验验收 Harness | 能跑但用户不会用 |
| 多设备表现 | 体验验收 Harness | 只在单一设备可用 |
| 上线检查 | 交付迭代 Harness | 未检查就交付 |
| 反馈分类 | 交付迭代 Harness | bug、新需求、体验问题混在一起 |
| 变更流程 | 交付迭代 Harness | 直接改代码，无法追溯 |

## 3. 检查步骤

第一步：列出 AI 可能产出的内容。

第二步：为每一项找到约束文档。

第三步：找不到约束时，先补模板或 checklist，再进入执行。

## 4. 合格标准

- 主要产出维度都有约束。
- 每个任务都有允许范围和禁止范围。
- 每个功能都有可检查标准。
- 每个视觉和交互决策都有设计依据。
- 每个字段和接口都有工程规格依据。
- 非技术人能根据文档判断是否通过。
- 问题能追溯到具体文档或 checklist。

## 5. 一句话原则

```text
AI 可能产出的内容，必须能被 harness 覆盖。
```
