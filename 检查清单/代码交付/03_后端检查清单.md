# 后端检查清单

## 1. 技术栈

- [ ] 使用 Java 17+
- [ ] 使用 Spring Boot 3
- [ ] 使用 Maven 或 Maven Wrapper
- [ ] 使用 Spring Web
- [ ] 使用 Spring Validation
- [ ] 使用 MyBatis-Plus 作为默认数据访问框架
- [ ] 使用 Flyway 管理数据库结构变更
- [ ] 明确不使用 Spring Cloud
- [ ] 后端是单体应用

## 2. 目录结构

- [ ] `controller/v1/` 只放 API v1 控制器
- [ ] `domain/*/entity/` 放数据库实体对象
- [ ] `domain/*/bo/` 放业务入参对象
- [ ] `domain/*/vo/` 放接口返回对象
- [ ] `mapper/` 放 MyBatis-Plus Mapper
- [ ] `service/` 放 Service 接口
- [ ] `service/impl/` 放 Service 实现
- [ ] `exception/` 放异常和错误响应
- [ ] `config/` 放配置
- [ ] `resources/db/migration/` 放 Flyway migration

## 3. 命名规范

- [ ] 已阅读 `文档/工程基线/06_前后端命名规范.md`
- [ ] Java 包名全部小写
- [ ] Controller 使用业务名 + Controller
- [ ] Entity 使用业务名
- [ ] BO 使用动作 + 业务名 + Bo
- [ ] VO 使用业务名 + Vo
- [ ] Mapper 使用业务名 + Mapper
- [ ] Service 接口使用 I + 业务名 + Service
- [ ] Service 实现使用业务名 + ServiceImpl
- [ ] API 路径使用小写 kebab-case
- [ ] 请求和响应字段使用 lowerCamelCase
- [ ] 数据库表和字段使用 snake_case
- [ ] 错误码使用 UPPER_SNAKE_CASE
- [ ] 没有为同一业务概念新增多个英文名

## 4. 数据库规范

- [ ] 已阅读 `文档/工程基线/07_数据库设计规范.md`
- [ ] 新表已先写入 `目标项目/docs/05_数据库结构.md`
- [ ] 新表有 Flyway migration
- [ ] 表名使用 snake_case
- [ ] 字段名使用 snake_case
- [ ] 每张业务表包含 7 要素：`id`、`created_by`、`created_at`、`updated_by`、`updated_at`、`is_valid`、`is_deleted`
- [ ] 主键字段为 `id`
- [ ] 主键类型为 `CHAR(32)`
- [ ] 后端 Entity、VO、Controller PathVariable、Service 入参统一使用 `String id`
- [ ] 不使用自增 BIGINT 主键
- [ ] 不使用 `Long id` 表达业务主键
- [ ] 创建人字段为 `created_by`，类型为 `CHAR(32)`
- [ ] 更新人字段为 `updated_by`，类型为 `CHAR(32)`
- [ ] 系统用户默认值为 `00000000000000000000000000000000`
- [ ] 更新时间字段为 `updated_at`
- [ ] 是否有效字段为 `is_valid`
- [ ] 是否删除字段为 `is_deleted`
- [ ] 时间字段默认由数据库 trigger 维护
- [ ] trigger 命名符合 `trg_表名_bi`、`trg_表名_bu`
- [ ] 多表查询默认使用主键 `id` 关联
- [ ] 关联字段命名符合 `业务对象_id`，类型为 `CHAR(32)`
- [ ] 状态字段使用 `status`
- [ ] 状态值已在接口契约或数据库结构文档中说明
- [ ] 主键、索引、非空字段已定义
- [ ] 唯一约束来自明确业务规则
- [ ] 索引命名使用 `idx_表名_字段名`
- [ ] 唯一索引命名使用 `uk_表名_字段名`
- [ ] 字段命名和接口契约可追溯
- [ ] Entity 字段和数据库字段可追溯
- [ ] VO 只暴露接口契约允许的字段
- [ ] 没有直接修改历史 migration

## 5. API 版本和多端

- [ ] 新接口默认使用 `/api/v1`
- [ ] 不兼容变更新增 `/api/v2`，不破坏 v1
- [ ] 多端共用接口契约
- [ ] 如需客户端类型，使用 `X-Client-Type`
- [ ] 客户端类型没有在业务逻辑中形成大量条件分支

## 6. 编码约束

- [ ] Controller 没有业务逻辑
- [ ] Controller 没有直接访问数据库
- [ ] Controller 只依赖 Service 接口
- [ ] Service 存在 interface 和 impl
- [ ] Service 没有处理 HTTP 细节
- [ ] Mapper 没有业务判断
- [ ] Entity 没有直接暴露给前端
- [ ] BO 有参数校验
- [ ] VO 只返回接口契约允许的字段
- [ ] 错误响应格式统一
- [ ] 配置不包含真实密钥
- [ ] 查询默认过滤 `is_valid = 1` 和 `is_deleted = 0`

## 7. 验证

当前后端模板没有提交 Maven Wrapper 时，使用：

- [ ] `mvn test` 可通过
- [ ] `mvn package` 可通过
- [ ] `mvn spring-boot:run` 可启动

如果目标项目补充了 Maven Wrapper，可以使用：

- [ ] `./mvnw test` 可通过
- [ ] `./mvnw package` 可通过
- [ ] `./mvnw spring-boot:run` 可启动

其他验证：

- [ ] API 返回符合契约
- [ ] 错误响应符合契约
- [ ] Service 接口和实现均有对应测试计划
- [ ] CRUD 样板可作为新业务模块参考
- [ ] Flyway migration 中 trigger 已在目标数据库验证可执行
