# 文档工程落地流程

## 1. 为什么要先有文档工程

本课程不是让 AI 直接生成代码，而是用文档把产品一步步约束清楚，再让 AI 或其他代码生成 Agent 按文档生成代码。

错误方式：

```text
我有一个想法
↓
直接让 AI 写代码
↓
代码看起来很多，但不知道为什么这样写
↓
出错后无法判断哪里错
```

正确方式：

```text
产品想法
↓
项目总说明
↓
产品需求文档
↓
页面清单
↓
用户流程
↓
数据规范
↓
技术结构
↓
任务清单
↓
文档到代码映射
↓
选择合适的代码生成 Agent
↓
Agent 按任务实现代码
↓
审查和验收
```

## 2. 文档工程和 Agent 的关系

本课程真正要教的不是“必须用某一个 AI 编码工具”，而是：

```text
文档工程是控制层。
Agent 是执行层。
```

文档工程负责：

- 定义产品边界。
- 定义页面和流程。
- 定义数据结构。
- 定义文件职责。
- 定义任务边界。
- 定义验收标准。

Agent 负责：

- 按任务生成代码。
- 按报错修复问题。
- 输出修改文件和验证方式。

因此，可以使用不同模型或 Agent 生成代码，但不能绕过文档工程。

## 3. 每份文档解决什么问题

| 文档 | 解决的问题 | 输出到哪里 |
|---|---|---|
| 项目总说明 | 这个产品到底是什么 | README、页面文案、范围边界 |
| 产品需求文档 | 第一版做什么、不做什么 | 功能清单、验收标准 |
| 页面清单 | 有哪些页面，每页做什么 | `index.html` 页面结构 |
| 用户流程 | 用户怎么完成核心动作 | `app.js` 交互逻辑 |
| 数据规范 | 页面需要哪些数据字段 | `mock/places.js` 数据结构 |
| 技术结构 | 文件怎么分、职责怎么分 | 文件目录和代码边界 |
| 任务清单 | Agent 每次改什么 | Agent 编码任务 |
| 文档到代码映射 | 哪个文档约束哪个文件 | 代码审查和验收 |
| Agent 实现指南 | 如何选择和约束 Agent | Agent 输入、分工、验收 |

## 4. H5 第一版为什么适合讲文档工程

H5 第一版只有 5 类核心文件：

```text
index.html       页面结构
styles.css       视觉样式
app.js           交互逻辑
mock/places.js   模拟数据
README.md        项目说明
```

这 5 类文件刚好能对应产品工程里的 5 个最小问题：

```text
页面上有什么？
页面长什么样？
用户点了会发生什么？
数据从哪里来？
怎么判断做完了？
```

## 5. 本样例的落地顺序

本样例应该按下面顺序推进，而不是直接写代码：

1. 写 `01_PROJECT_BRIEF.md`。
2. 写 `02_PRD.md`。
3. 写 `03_PAGE_LIST.md`。
4. 写 `04_UX_FLOW.md`。
5. 写 `05_DATA_SPEC.md`。
6. 写 `06_ARCHITECTURE.md`。
7. 写 `07_TASKS.md`。
8. 写 `08_DOC_TO_CODE_MAPPING.md`。
9. 写 `09_AGENT_IMPLEMENTATION_GUIDE.md`。
10. 选择合适的代码生成 Agent。
11. 让 Agent 按 `07_TASKS.md` 实现代码。
12. 用 `08_DOC_TO_CODE_MAPPING.md` 和任务验收标准做审查。

## 6. 课程里的关键判断

只要学员看不到“文档如何约束代码”，就说明课程没有讲完整。

所以每次新增代码前，都必须回答：

```text
这段代码来自哪份文档？
它实现了哪个功能？
它属于哪个页面？
它使用了哪个数据字段？
它满足哪个验收标准？
是哪个 Agent 执行的？
这个 Agent 是否遵守了任务边界？
```
