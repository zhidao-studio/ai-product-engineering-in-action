# 需求实现控制检查清单

这份清单用于判断：当前需求是否已经被文档控制住，是否可以交给 Agent 实现，以及实现后是否可以验收通过。

## 1. 实现前检查

### 1.1 产品边界

- [ ] 需求是否写入 `01_PROJECT_BRIEF.md`？
- [ ] 目标用户是否明确？
- [ ] 第一版做什么是否明确？
- [ ] 第一版不做什么是否明确？
- [ ] 是否没有把登录、后端、地图、支付等复杂能力偷偷加回来？

### 1.2 功能范围

- [ ] 功能是否写入 `02_PRD.md`？
- [ ] 功能是否有编号？
- [ ] 功能是否有优先级？
- [ ] 功能是否有验收标准？
- [ ] 是否能判断这个功能属于第一版？

### 1.3 页面控制

- [ ] 页面是否写入 `03_PAGE_LIST.md`？
- [ ] 页面入口是否明确？
- [ ] 页面目标是否明确？
- [ ] 页面状态是否明确？
- [ ] 页面数量是否没有失控？

### 1.4 流程控制

- [ ] 用户动作是否写入 `04_UX_FLOW.md`？
- [ ] 用户动作后是否有结果承接？
- [ ] 核心流程是否闭环？
- [ ] 是否能完成：首页 → 详情 → 收藏 → 收藏页查看？

### 1.5 数据控制

- [ ] 数据字段是否写入 `05_DATA_SPEC.md`？
- [ ] 字段名是否统一？
- [ ] 字段类型是否明确？
- [ ] 收藏数据结构是否明确？
- [ ] 是否禁止 Agent 自行发明字段？

### 1.6 技术边界

- [ ] 文件职责是否写入 `06_ARCHITECTURE.md`？
- [ ] 是否明确不引入框架？
- [ ] 是否明确不接入后端？
- [ ] 是否明确不接入地图？
- [ ] 是否明确每个文件负责什么？

### 1.7 Agent 任务边界

- [ ] 当前任务是否写入 `07_TASKS.md`？
- [ ] 当前任务是否只做一件事？
- [ ] 是否写清允许修改文件？
- [ ] 是否写清禁止修改文件？
- [ ] 是否写清完成后输出格式？

## 2. Agent 输入检查

把任务交给 Agent 之前，输入必须包含：

- [ ] 当前任务编号。
- [ ] 当前任务目标。
- [ ] 必须阅读的文档。
- [ ] 允许修改的文件。
- [ ] 禁止修改的文件和能力。
- [ ] 验收标准。
- [ ] 完成后输出要求。

如果缺任何一项，不要让 Agent 开始写代码。

## 3. 实现后检查

### 3.1 任务边界检查

- [ ] Agent 是否只完成当前任务？
- [ ] Agent 是否只修改允许文件？
- [ ] Agent 是否没有修改无关文件？
- [ ] Agent 是否没有新增未经允许的依赖？
- [ ] Agent 是否没有扩大产品范围？

### 3.2 文档映射检查

- [ ] 代码是否能对应到 `08_DOC_TO_CODE_MAPPING.md`？
- [ ] 页面实现是否来自页面清单？
- [ ] 交互实现是否来自用户流程？
- [ ] 数据字段是否来自数据规范？
- [ ] 文件结构是否符合技术结构？

### 3.3 功能验收检查

- [ ] 首页能显示地点列表。
- [ ] 点击地点能进入详情页。
- [ ] 详情页能显示地点信息。
- [ ] 收藏按钮能切换状态。
- [ ] 收藏页能显示已收藏地点。
- [ ] 设置页能清空收藏。
- [ ] 关于页能说明课程样例定位。

### 3.4 用户验收检查

- [ ] 用户第一次打开能看懂这是做什么的。
- [ ] 用户知道下一步点哪里。
- [ ] 用户能完成核心流程。
- [ ] 用户动作后能看到结果。
- [ ] 用户知道当前是课程样例数据。

## 4. 需求变更检查

如果出现新需求，先判断：

- [ ] 是否属于第一版范围？
- [ ] 是否需要修改 PRD？
- [ ] 是否需要修改页面清单？
- [ ] 是否需要修改用户流程？
- [ ] 是否需要修改数据规范？
- [ ] 是否需要新增任务？

不允许直接让 Agent 改代码。

## 5. 最终结论

每次实现后，必须给出一个结论：

```text
是否通过：通过 / 有条件通过 / 不通过
原因：
必须修复：
可选优化：
是否可以进入下一任务：是 / 否
```

## 6. 一句话原则

```text
需求没有进文档，就不能进代码。
代码不能回溯到文档，就不能通过验收。
```
