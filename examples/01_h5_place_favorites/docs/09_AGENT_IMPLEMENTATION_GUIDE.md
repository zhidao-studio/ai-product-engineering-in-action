# Agent 实现指南

## 1. 先回答核心问题

当然可以用别的模型或 Agent 来生成代码。

本课程真正要教的不是“必须用某一个 AI 编码工具”，而是：

```text
用文档工程控制产品边界，
用任务清单约束 Agent 行为，
用验收标准判断 Agent 产物。
```

也就是说：

```text
文档工程是控制层。
Agent 是执行层。
```

只要 Agent 能遵守文档、按任务修改文件、输出修改说明和验证方式，就可以使用。

## 2. 为什么不是指定一个 Agent

不应该强绑定某个 Agent，原因是：

1. 不同学员能使用的工具不同。
2. AI 编码工具变化很快。
3. 有些 Agent 擅长写前端，有些擅长改工程，有些擅长审查。
4. 课程要教的是可迁移的方法，而不是某个工具按钮。

所以课程应该表达为：

```text
先建立文档工程，
再选择任意合适的代码生成 Agent 执行任务。
```

## 3. 什么样的 Agent 可以用

只要满足下面条件，就可以用于本样例：

| 条件 | 说明 |
|---|---|
| 能读取多个文档 | 至少能读取项目总说明、PRD、页面清单、任务清单 |
| 能修改工程文件 | 能创建或修改 `index.html`、`styles.css`、`app.js`、`mock/places.js` |
| 能遵守修改范围 | 不乱改无关文件 |
| 能说明改了什么 | 输出修改文件列表和实现说明 |
| 能按报错修复 | 只修当前问题，不大范围重写 |
| 能接受验收标准 | 按任务里的验收标准完成 |

## 4. Agent 输入必须来自文档

不要这样输入：

```text
帮我做一个 H5 地点收藏页面。
```

应该这样输入：

```text
请阅读以下文档：

- 01_PROJECT_BRIEF.md
- 02_PRD.md
- 03_PAGE_LIST.md
- 04_UX_FLOW.md
- 05_DATA_SPEC.md
- 06_ARCHITECTURE.md
- 07_TASKS.md
- 08_DOC_TO_CODE_MAPPING.md

当前只执行 TASK-001。

目标：
建立 H5 基础文件结构。

允许修改：
- index.html
- styles.css
- app.js
- mock/places.js

禁止修改：
- 不引入框架
- 不接入后端
- 不接入地图
- 不新增登录
- 不做任务外功能

验收标准：
浏览器能打开基础页面，并能看到课程样例说明。

完成后输出：
1. 修改文件列表
2. 实现说明
3. 验证方式
4. 风险点
```

## 5. 多 Agent 分工方式

可以使用多个 Agent，但必须分工清楚。

推荐分工：

| Agent 类型 | 负责什么 | 不负责什么 |
|---|---|---|
| 产品/文档 Agent | 生成和修正文档 | 不直接改代码 |
| 代码生成 Agent | 按任务实现代码 | 不扩大产品范围 |
| 审查 Agent | 检查代码是否符合文档 | 不重写全部代码 |
| 修复 Agent | 根据报错修复当前问题 | 不做无关优化 |

## 6. 不建议的方式

不要让多个 Agent 同时改同一个任务。

错误方式：

```text
让 Agent A、Agent B、Agent C 同时实现首页。
```

问题：

- 文件容易冲突。
- 风格不一致。
- 学员不知道采用哪个版本。
- 问题出现后难以定位。

正确方式：

```text
一个任务只交给一个代码生成 Agent。
完成后交给另一个审查 Agent 审查。
```

## 7. Agent 产物怎么验收

验收时不看 Agent 说“完成了”，而看它是否满足文档约束：

```text
是否对应当前 TASK？
是否只改允许文件？
是否实现 PRD 功能？
是否符合页面清单？
是否符合用户流程？
是否使用数据规范字段？
是否满足验收标准？
```

## 8. 本课程的真正抽象

本课程的可迁移结构是：

```text
文档工程
↓
任务约束
↓
Agent 实现
↓
代码审查
↓
模拟用户验收
```

其中 Agent 可以替换。

不可替换的是：

```text
需求边界
任务边界
验收标准
人工判断
```

## 9. 一句话结论

```text
可以用任何合适的模型或 Agent 生成代码，
但不能绕过文档工程、任务约束和验收标准。
```
