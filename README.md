# 文档驱动的 AI 产品交付方法论

Document-Driven AI Product Delivery Harness

## 1. 仓库定位

本仓库用于沉淀一套阶段受控的 AI 产品交付协议。

它的核心目标是：

```text
用文档表达意图，
用结构化契约收束歧义，
用工程模板固定边界，
用检查清单和机器门禁阻止 AI 越界，
最终让 AI 从产品想法走向可验收的代码交付。
```

## 2. 先读总览

为避免仓库越加越散，先读：

```text
文档/控制系统/00_系统总览.md
```

这个文件说明当前仓库是什么、分几层、哪些内容不要再加、下一步只做什么。

## 3. 核心链路

```text
产品定义
↓
界面规格
↓
高保真交互预览
↓
使用者视觉确认
↓
工程规格
↓
工程基线
↓

↓
代码任务包
↓
AI 在模板工程内编码
↓
自动化验证
↓
使用验收
↓
交付迭代
```

## 4. 当前只分四层

```text
文档层：表达人的意图和已确认决策
契约层：把关键约束结构化，减少 AI 自由解释
门禁层：用脚本或检查阻止 AI 越界
反馈层：把技术结果翻译成人能决策的语言
```

后续新增内容必须归入这四层之一，否则优先合并到已有文件。

## 5. 仓库结构

```text
文档/        方法论、工程基线、控制系统、操作手册、教学化内容
样板/        可复用空白样板、控制系统元数据样板

检查清单/    可执行检查清单
指令/        AI / Agent / Review 指令
案例/        填好的案例
```

## 7. 当前工程基线

```text
Web 后台：Vue 3 + Vite + TypeScript + Vue Router + Pinia + Element Plus
H5：Vue 3 + Vite + TypeScript + Vant
小程序：原生微信小程序 + TypeScript + Vant Weapp
iOS：SwiftUI + MVVM
后端：Java + Spring Boot + MyBatis-Plus + Flyway 单体应用，不使用 Spring Cloud
数据库：MySQL 优先；主键 32 位字符串；每表 7 要素；时间 trigger 维护
码值：数据库码值是源头，后端常量一致，接口契约记录，前端只翻译不改码
部署：本地开发优先，微信云托管按需适配
多端原则：多个工程，独立建设；接口一致，代码不共享
```
