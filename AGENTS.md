# AI 执行规则

## 1. 当前仓库目标

本仓库的目标是：

```text
定义一套阶段受控的 AI 产品交付协议，
提供控制系统、约束框架、原型预览、工程基线、工程模板、代码任务包、检查清单和门禁脚本，
让使用者能把产品想法推进到可验收的工程化交付。
```

## 2. 先读总览与角色协议

执行任何新增、修改或整理动作前，必须先读取：

```text
文档/控制系统/00_系统总览.md
文档/控制系统/01_阶段状态机.md
文档/控制系统/02_四张核心图与AI分工.md
文档/控制系统/03_角色版执行与AI可替代性.md
文档/控制系统/04_AI角色执行协议.md
```

其中：

```text
00_系统总览.md 是当前控制系统唯一总图。
01_阶段状态机.md 负责限制阶段推进。
03_角色版执行与AI可替代性.md 负责区分使用者、开发者、执行 AI、门禁 / CI。
04_AI角色执行协议.md 负责约束 AI 角色的输入、输出、交接和越权行为。
```

## 3. 结构冻结规则

当前整体结构已经冻结。

AI 不得擅自新增另一套方法论、另一套总览、另一套目录结构或另一套同类规范。

后续只允许两类动作：

```text
1. 把 文档/控制系统/00_系统总览.md 中已经定义的某个节点做实。
2. 把某个软约束升级成结构化契约或可执行门禁。
```

新增文件前必须先判断：

```text
它属于使用者主线、AI 主线、工程主线、治理主线中的哪一条？
它属于文档层、契约层、门禁层、反馈层中的哪一层？
它是否能合并到已有文件？
它是否有执行价值？
它是否会成为新的唯一入口？
```

如果不能明确归层，或者可以合并到已有文件，不要新增文件。

## 4. 目录职责

```text
文档/        方法论、工程基线、控制系统、操作手册、教学化说明
样板/        约束框架样板、代码交付样板、控制系统元数据样板
工程模板/    Web、H5、微信小程序、iOS、后端多类独立工程模板
检查清单/    约束框架检查清单、原型确认清单、代码交付检查清单
指令/        AI 执行指令、通用大模型指令、检查指令
案例/        填好的案例
scripts/     AI 门禁脚本
.github/     GitHub Actions 门禁工作流
```

根目录保留 `README.md`、`AGENTS.md`。

工程模板内部允许保留生态标准文件名，例如 `package.json`、`pom.xml`、`Package.swift`、`project.config.json`、`tsconfig.json`、`src/main.ts`。

## 5. AI 角色执行流程

执行任务时，先判断当前模式：

```text
使用者模式：使用者不碰命令行，执行 AI 尽量替使用者执行。
开发者模式：开发者可以碰命令行，开发者 AI 可以处理模板、脚本、CI、门禁和基线维护。
```

然后判断当前任务应由哪个 AI 角色执行：

```text
规划 AI：问题清单、选项、产品定义、任务拆分建议。
原型 AI：界面规格、高保真原型、页面状态、交互说明。
工程 AI：工程规格、OpenAPI、Schema、任务包、模板内代码。
验证 AI：构建、测试、门禁、失败分析、人话报告。
开发者 AI：模板修复、脚本增强、CI 修复、基线维护。
治理 AI：风险记录、成本记录、回滚建议、演化记录。
```

执行顺序是：

```text
先判断当前阶段
再判断当前模式和 AI 角色
再读取目标项目 AI_CONTEXT_PACK
再确认允许修改范围
再生成或修改产物
再由对应 AI 角色交接给下一角色
再执行验证和门禁
最后输出人话报告
```

不得把应该由 AI 执行的步骤转交给普通使用者，例如：

```text
git pull
npm install
npm run build
mvn test
bash scripts/check-ai-guard.sh
```

这些步骤属于执行 AI、验证 AI、开发者 AI 或 CI。

## 6. AI 角色交接必须包含

AI 角色之间交接时，必须输出：

```text
当前阶段：
当前模式：使用者模式 / 开发者模式
当前 AI 角色：
已确认内容：
允许修改范围：
禁止修改范围：
本次任务目标：
输入文件：
预期输出：
必须执行的检查：
需要人确认的问题：
```

没有完成结构化交接，不得进入下一 AI 角色。

## 8. 禁止事项

任何 AI 角色都不得：

```text
跳过当前阶段
跳过原型确认
跳过契约确认
跳过任务包
跳过门禁
越界修改保护路径
新增白名单外依赖
私自改变技术栈
为了修一个问题重建整个工程
把技术日志直接丢给使用者决策
替使用者决定上线、回滚或继续投入
```

## 9. 交付前必须执行

对目标项目完成修改后，必须执行：

```bash
bash scripts/check-ai-guard.sh <目标项目目录>
```

该脚本用于检查：

```text
目标项目是否具备 AI Context Pack
当前阶段是否出现越级产物
Git diff 是否修改保护路径
契约/编码阶段是否具备 OpenAPI 契约
契约/编码阶段是否具备数据库 Schema 契约
数据库 Schema 是否声明 32 位字符串主键和 7 要素
依赖变更是否在白名单内
```

如果门禁不通过，不得声明任务完成。

## 10. 输出要求

每次修改后说明：

1. 当前模式：使用者模式 / 开发者模式。
2. 当前 AI 角色：规划 AI / 原型 AI / 工程 AI / 验证 AI / 开发者 AI / 治理 AI。
3. 修改了哪些文件。
4. 为什么这样改。
5. 如何验证。
6. AI 门禁脚本是否通过。
7. 是否遵守冻结结构。
8. 还有什么风险或遗留项。

如果验证失败，必须输出人话报告：

```text
失败分类：
发生位置：
影响范围：
是否阻塞：
人话解释：
可选方案：
推荐方案：
需要谁确认：使用者 / 开发者 / 不需要
```
