# 接口契约样板

## 1. 目标

本文件用于约束多端和后端如何通信。

AI 不得自行发明接口路径、请求参数、响应字段或错误码。

## 2. 基础约定

接口前缀：`/api/v1`

数据格式：JSON

认证方式：待填写

分页方式：待填写

默认调用端：Web 后台 / H5 / 微信小程序 / iOS / 其他

## 3. API 版本规则

新增接口默认进入：

```text
/api/v1/...
```

不兼容变更必须新增版本，例如：

```text
/api/v2/...
```

不得直接破坏旧版本接口的请求字段、响应字段、错误码和语义。

字段废弃时必须先标记 deprecated，并在交付迭代记录中说明。

## 4. 多端接入规则

后端接口默认面向多端接入。

可选客户端类型请求头：

```text
X-Client-Type: web-admin / h5 / weapp / ios
```

客户端类型主要用于日志、审计、兼容判断和灰度判断。

不允许在业务逻辑中大量散落端类型条件分支。

端差异优先通过：

```text
通用 DTO
查询参数
视图模式字段
API 版本
```

处理。

## 5. 认证规则

如果项目需要登录，必须明确：

| 项目 | 约定 |
|---|---|
| 登录态保存方式 | 优先 HttpOnly Cookie / Bearer Token / 其他 |
| access token 有效期 | 例如 30 分钟 |
| refresh token 机制 | 是否需要，如何刷新 |
| 401 响应后前端行为 | 跳转登录 / 提示登录失效 / 尝试刷新 |
| 是否携带凭证 | Cookie 模式下必须明确 `credentials` 策略 |
| CSRF 防护 | Cookie 模式下必须说明 CSRF token 或 SameSite 策略 |

默认建议：

```text
内部 MVP 优先使用 HttpOnly Cookie + SameSite + CSRF 防护。
如必须使用 Bearer Token，必须在任务包中说明 token 存储风险。
```

## 10. 错误码

| 错误码 | HTTP 状态 | 说明 | 前端展示 |
|---|---:|---|---|
| UNAUTHORIZED | 401 | 未登录或登录失效 | 按认证规则处理 |
| FORBIDDEN | 403 | 无权限 | 展示无权限提示 |
| INVALID_REQUEST | 400 | 参数错误 | 展示字段或表单错误 |
| INTERNAL_ERROR | 500 | 服务异常 | 展示系统错误提示 |
